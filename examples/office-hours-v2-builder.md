# 示例：Office Hours v2 — Builder 模式

## 用户输入

```text
@砺 我想周末做个好玩的网页：输入一句话，生成一个“今天适合当什么动物”的小测试。不打算商业化，就是想做个能分享的 demo。
```

## 模式判断

```markdown
- 模式：Builder
- 理由：用户明确说“周末”“好玩”“demo”“不打算商业化”
- 砺的目标：保护创造性，帮用户找到最快可分享路径，而不是强行商业拷问
```

## Builder 5 问

1. 这个东西最酷的版本是什么？
2. 你会给谁看这个？什么会让他们说“哇”？
3. 到达一个你能真正使用或分享的东西，最快路径是什么？
4. 最接近的现有产品是什么？你的差异在哪里？
5. 如果时间无限，你会加什么？10x 版本是什么？

## Builder Design Doc 骨架

```markdown
# 今天适合当什么动物 — Builder Design Doc

- Status: Draft
- Mode: Builder
- Related Docs: 无

## Coolest Version
用户输入一句话后，页面用动物人格 + 一句吐槽 + 分享卡片形成惊喜。

## Intended Audience
朋友圈 / 小红书熟人社交用户；目标反应是“这也太像我了”。

## Fastest Shareable Path
单页网页：输入框 → 生成结果 → 分享卡片截图。

## Closest Existing Thing
MBTI、小测试、抽签网页。差异是“当天状态”而非长期人格。

## 10x Version
每日动物状态日历 + 好友互测 + 连续签到生成动物迁徙图。

## Prototype Plan
1. 先写 12 个动物结果模板。
2. 前端随机 / 轻量规则匹配。
3. 生成结果卡片。
4. 发给 10 个朋友测试分享欲。

## Risks / Non-goals
- 不做账号系统。
- 不做复杂 AI。
- 不追求心理学严谨。

## What I Noticed
用户的目标是“可分享的惊喜”，不是严肃测试；最快路径应优先结果文案和卡片效果。
```

## 下游交接

- 需要页面草图 → 交 `绘`。
- 需要实现 → 交 `铸`。
- 如果测试后出现真实增长 / 付费线索，再升级为 Startup 模式。
