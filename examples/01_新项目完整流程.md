# 示例：宠物健康管理 App — 完整流程演示

> 本示例展示如何用维弈阁团队从"一个想法"走到"可执行的开发计划"。

---

## 背景

你有一个想法：做一个 AI 驱动的宠物健康管理 App。下面演示如何用维弈阁团队评估和规划这个项目。

---

## Step 1: 质疑想法（@砺）

**你说**：
```
@砺 我有个想法：做一个 AI 驱动的宠物健康管理 App，
用户拍照上传宠物的症状，AI 给出初步诊断建议。你觉得怎么样？
```

**砺的回应**（YC 办公时间模式）：
```
在我们深入之前——你这次的目标是什么？

- 创业 — 正在做或考虑做一个创业项目
- 黑客马拉松/Demo — 时间限定，需要让人惊艳
- 好玩 — 副项目、创意出口
```

**你说**：创业。

**砺开始逐个追问**：

> 问题 1: 你有什么最强证据证明有人真正需要这个？

**你说**：我养了两只猫，每次生病都要花 500 块挂号费，结果医生看 5 分钟就说没事。身边养宠的朋友都有同样的痛点。

> 问题 2: 你的用户现在是怎么解决这个问题的？

**你说**：百度搜症状、问宠物群、直接去医院。

**砺的判断**：
```
⚠️ 有条件通过。

需求真实性：✅ 有个人痛点 + 朋友验证，但样本太小
现状替代方案：百度搜索是免费的，你的 App 凭什么比百度好？
最窄楔子：先做"拍照识别常见症状"，不做诊断，只做分诊建议（去不去医院）

建议：先做 10 个养宠人的访谈，验证付费意愿，再决定全量投入。

📤 交接块（Handoff）
- 来源: 合伙人_砺
- 状态: 有条件通过
- 关键决策: 先验证付费意愿再全量投入
- 下游建议: 交 @锋 做方向确认
```

---

## Step 2: 方向确认（@锋）

**你说**：
```
@锋 砺有条件通过了宠物健康 App 的想法，建议先验证付费意愿。
帮我做一下战略审查。
```

**锋的回应**（Quick 模式）：
```
✅ 方向可行。理由：

1. 宠物医疗是千亿市场，增长 20%/年
2. AI 图像识别技术成熟，成本可控
3. 竞品（萌爪医生等）验证了需求存在，但体验差

⚠️ 调整建议：
- 不做"AI 诊断"（医疗风险太高），做"AI 分诊"（告诉你要不要去医院）
- MVP：拍照 → AI 分析 → 建议（去医院/观察/没事）
- 先做猫，不做狗（聚焦）

📤 交接块（Handoff）
- 来源: CEO_锋
- 状态: 通过
- 关键决策: 定位为"AI 分诊"而非"AI 诊断"；先做猫
- 下游建议: 交 @枢 写 PRD
```

---

## Step 3: 写 PRD（@枢）

**你说**：
```
@枢 方向确定了：AI 宠物分诊 App，先做猫。帮我写 PRD。
```

**枢的回应**（消歧义 + PRD）：

枢先进行 5 层追问，然后产出 PRD：

```markdown
# 喵诊 — AI 猫咪分诊助手 PRD

**版本**: V1.0 | **状态**: 草稿

## 0. 一句话描述
拍照上传猫咪症状，AI 告诉你要不要去医院。

## 1. 问题 & 假设
- 问题：养猫人无法判断猫咪症状是否需要就医
- 核心假设：用户愿意为"省一次不必要的挂号费"付费
- 最危险假设：AI 分诊准确率能否达到用户信任阈值

## 2. 目标用户
- 25-35 岁一线城市养猫女性
- Jobs-to-be-Done：快速判断猫咪是否需要就医

## 3. MVP 定义
- 最小闭环：拍照 → AI 分析 → 三级建议（立即就医/观察 3 天/没事）
- 验证假设：用户是否信任 AI 的分诊建议
- 成功指标：首周留存 > 30%，分诊准确率 > 80%

## 4. 功能清单（MVP）
| 功能 | 用户故事 | 验收标准 | 优先级 |
|------|---------|---------|--------|
| 拍照上传 | 作为养猫人，我想拍照上传猫咪症状 | 支持拍照+相册 | P0 |
| AI 分诊 | 作为养猫人，我想知道要不要去医院 | 3 级建议+置信度 | P0 |
| 历史记录 | 作为养猫人，我想看之前的分诊记录 | 按时间倒序 | P1 |

## 5. NOT in scope
- ❌ 在线问诊（V2.0）
- ❌ 狗的分诊（V1.5）
- ❌ 付费功能（先免费验证）

📤 交接块（Handoff）
- 来源: PM_枢
- 状态: 通过
- 下游建议: 交 @矩 做架构评审，交 @绘 做 UI 设计
```

---

## Step 4: 架构评审（@矩）

**你说**：
```
@矩 PRD 写好了，帮我做架构评审。
```

**矩的回应**：
```
架构建议：

┌─────────┐     ┌──────────┐     ┌──────────┐
│ 移动端   │────▶│ API 网关  │────▶│ AI 服务   │
│ (React  │     │ (Node)   │     │ (Python) │
│  Native)│     └────┬─────┘     └──────────┘
└─────────┘          │
                ┌────▼─────┐
                │ PostgreSQL│
                └──────────┘

推荐方案：
- 前端：React Native（跨平台，一套代码）
- 后端：Node.js + Express（快速开发）
- AI：Python + FastAPI（调用视觉模型）
- 数据库：PostgreSQL

理由：默认无聊原则——全部用已证实的技术，创新代币留给 AI 模型。

📤 交接块（Handoff）
- 来源: 架构_矩
- 状态: 通过
- 下游建议: 可以开始开发。建议开发完成后交 @鉴 测试。
```

---

## 总结

通过 4 步，我们完成了从"一个想法"到"可执行的开发计划"：

```
@砺 质疑想法 → @锋 确认方向 → @枢 写 PRD → @矩 架构评审 → 开始开发
```

整个过程中：
- 每个 Agent 只做自己擅长的事
- 交接块确保信息不丢失
- 决策有记录、有理由

---

*这就是维弈阁的工作方式：一个人，一个 AI 团队，10x 生产效率。*
