# 示例：Office Hours v2 — Startup 模式

## 用户输入

```text
@deep @砺 我有个想法：做一个宠物情绪识别 App。用户上传宠物照片，AI 判断它现在开心、焦虑还是不舒服，并给建议。帮我判断值不值得做。
```

## 砺的工作方式

### 1. Context Summary

```markdown
## Context Summary
- 当前目标：判断宠物情绪识别 App 是否值得进入 PRD
- 项目阶段：pre-product
- 已读文件：README.md、TODO.md、PROTOCOL.md、ROUTER.md、合伙人_砺/SOUL.md
- Git 状态：有未提交文档变更；不影响构思判断
- 历史相似设计：未发现同名设计；可参考宠物人格测试方向
- 已有证据：用户尚未提供真实需求证据
- 明显风险：AI 判断准确性、宠物健康误导、用户是否愿意为此付费
- 需要用户补充的 1 个关键问题：你有什么最强证据证明宠物主真的需要“情绪识别”，而不是只觉得好玩？
```

### 2. Startup 六问节奏

砺逐个问，不一次性抛出全部：

1. 你有什么最强证据证明有人真正需要这个？
2. 用户现在怎么解决这个问题？那个替代方案花了他们多少成本？
3. 说出最需要这个产品的那个人的名字。他是谁？为什么现在就需要？
4. 最小可行版本是什么——这周就能让一个真实用户愿意使用、转发或付费？
5. 你有没有真的观察某人在不被指导的情况下完成这件事？
6. 如果 3 年后世界明显变化，这件事会更重要还是更不重要？

### 3. PREMISES

```markdown
## PREMISES
| # | 前提陈述 | 脆弱度 | 验证方法 | 如果错了会怎样 |
|---|---------|--------|---------|----------------|
| P1 | 宠物主愿意基于照片获取情绪判断 | 高 | 小红书落地页 + 20 人访谈 + 上传转化率 | 产品只剩娱乐属性 |
| P2 | AI 输出不会造成健康误导 | 高 | 文案安全审查 + 风险提示 + 不做医疗判断 | 有安全/合规风险 |
| P3 | 用户会转发结果卡片 | 中 | 3 个结果卡 A/B 测试分享率 | 获客成本上升 |
```

### 4. 推荐路径

- **最小可行版**：不做“医疗级情绪识别”，先做“宠物心情小测试 + 照片风格化结果卡”。
- **验证指标**：上传率、结果页完成率、分享率、愿意留下微信/邮箱的比例。
- **放弃条件**：100 个访问里上传率 < 10%，分享率 < 3%，且访谈无强痛点。

### 5. 下游交接

- UI 类想法 → 交 `绘` 做 Visual Sketch。
- 重大方向 → 交 `隐` 二评，再交 `锋` 拍板。
- 通过后 → 交 `枢` 写 PRD。
