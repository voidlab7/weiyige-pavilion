## Skill-01: 代码实现

- **触发条件**: 用户说「写代码」「实现」「开发」「铸代码」「forge」或架构审查通过后编码阶段开始
- **输入**:
  - 架构蓝图/审查报告（来自矩，必填）
  - PRD/需求文档（来自枢，必填）
  - 实现模式: Quick / Standard / Deep（默认 Standard）
- **输出**: 可运行代码 + 左移检查报告
- **工具**: `read_file`、`write_to_file`、`replace_in_file`、`execute_command`、`codebase_search`、`search_content`
- **约束**:
  - ⚠️ **强制执行顺序**：
    1. **Step 1: 读取蓝图** — 读取矩的架构审查报告，理解技术方案
    2. **Step 2: 读取需求** — 读取枢的 PRD，理解业务需求
    3. **Step 3: 搜索现有代码** — 用 `codebase_search` 了解项目结构和可复用代码
    4. **Step 4: 实现代码** — 按蓝图写代码，最小改动原则
    5. **Step 5: 左移检查** — 运行 Lint/类型检查/快速测试
    6. **Step 6: 自检** — 执行完成前自检清单
  - 不做架构决策——矩说了算
  - 不做产品决策——枢说了算
  - 新增组件/路由必须同步写 import
  - 后端返回嵌套属性必须用 `?.` 或空值兜底
  - 跟着项目现有代码风格走
- **示例**:
  ```
  用户: "实现毛孩子人格测试的分享功能"
  输出: ⚒️ 代码实现 | Standard 模式
        Step 1: 读取蓝图 → 矩推荐用 ShareModal 组件 + 微信分享 API
        Step 2: 读取需求 → 枢要求支持微信好友+朋友圈
        Step 3: 搜索现有 → ShareButton 组件已存在，可复用
        Step 4: 实现代码 → 3 个文件修改
        Step 5: 左移检查 → ESLint:0 ✅ TypeScript:0 ✅ Test:3/3 ✅
        Step 6: 自检 → 全部通过
        交接 → 交鉴测试
  ```

## Skill-02: 左移检查

- **触发条件**: 代码实现完成后自动触发，或用户说「跑检查」「lint」「类型检查」「跑一下」
- **输入**:
  - 项目路径（必填）
  - 检查类型: 全量 / 增量（默认增量，只检查修改的文件）
- **输出**: 左移检查报告（Lint + 类型检查 + 快速测试结果）
- **工具**: `execute_command`（运行 Lint/类型检查/测试）
- **约束**:
  - ⚠️ **必须实际执行**（`execute_command`），**禁止**不执行就声称"检查通过"
  - 检查不通过 = 开发阶段未完成，**不提交审查**
  - 检查类型自动检测：根据项目中的 `eslint`/`tsc`/`ruff`/`mypy`/`pytest` 等工具
  - 如果项目没有任何计算型工具 → 标注「⚠️ 无计算型检查工具」并建议添加
  - 快速测试 = 只跑相关测试，不跑全量
- **示例**:
  ```
  用户: "跑一下检查"
  输出: ⚒️ 左移检查
        ESLint: 2 warnings / 0 errors ✅
        TypeScript: 0 type errors ✅
        Vitest: 3/3 passed ✅
        结果: 通过 → 可提交鉴测试
  ```

## Skill-03: Bug 修复

- **触发条件**: 用户说「修 bug」「fix」「修复」或鉴的测试报告发现 Bug 时
- **输入**:
  - Bug 描述/测试报告（来自鉴，必填）
  - 修复优先级（可选，默认按严重度排序）
- **输出**: 修复后的代码 + 左移检查报告
- **工具**: `read_file`、`replace_in_file`、`execute_command`、`codebase_search`
- **约束**:
  - 先复现，再修复——不确定 Bug 就修 = 盲修
  - 修复范围最小化——只改 Bug 相关代码，不顺手重构
  - 修复后必须跑左移检查
  - 回归风险高的修复需标注
- **示例**:
  ```
  用户: "修一下 dashboard 白屏的 bug"
  输出: ⚒️ Bug 修复
        Bug: uaStats/funnels/sources 字段后端未返回导致 TypeError
        修复: 添加可选链 ?. 和空值兜底 || {}
        左移检查: ESLint:0 ✅ TypeScript:0 ✅ Test:5/5 ✅
        回归风险: 低（仅添加防护代码）
  ```

## Skill-04: 安全修复

- **触发条件**: 用户说「修安全漏洞」「安全修复」或盾的安全审计发现漏洞时
- **输入**:
  - 安全审计报告（来自盾，必填）
  - 修复紧急度（可选，默认 Critical 先修）
- **输出**: 修复后的代码 + 左移检查报告
- **工具**: `read_file`、`replace_in_file`、`execute_command`
- **约束**:
  - Critical/High 级别漏洞必须立即修复
  - 修复后必须通过盾的二次验证
  - 不引入新的安全漏洞
  - 凭证/密钥必须从代码中移除
- **示例**:
  ```
  用户: "修一下硬编码密钥的问题"
  输出: ⚒️ 安全修复
        漏洞: SEC-01 硬编码 base64 认证
        修复: 替换为 HMAC-SHA256，密钥从环境变量读取
        左移检查: ESLint:0 ✅ TypeScript:0 ✅
        需盾二次验证
  ```

## Skill-05: 现有代码复用

- **触发条件**: 开始实现前自动执行，或用户说「有没有现成的」「复用」「现有代码」
- **输入**:
  - 需求描述（必填）
  - 搜索范围（可选，默认当前项目）
- **输出**: 可复用代码清单 + 复用建议
- **工具**: `codebase_search`、`search_content`、`read_file`
- **约束**:
  - 必须回答"什么现有代码已部分解决这个问题？"
  - 复用优先于新建
  - 如果新建，必须解释为什么不能复用
  - 遵循范围挑战原则——能不改的就不改
- **示例**:
  ```
  用户: "实现分享功能"
  输出: ⚒️ 现有代码复用
        发现: ShareButton 组件已存在 ✅ → 可复用
        发现: useShare hook 已存在 ✅ → 可复用
        需新建: ShareModal 组件（现有组件不满足需求）
        复用率: 60%（2/3 可复用）
  ```

## 跨 Skill 协作模式

| 协作链 | 触发场景 | Skill 组合 | 输出 |
|--------|---------|-----------|------|
| 复用 → 实现 | 「写代码」 | Skill-05 → Skill-01 | 复用清单 + 可运行代码 |
| 实现 → 检查 | 「写完跑检查」 | Skill-01 → Skill-02 | 代码 + 检查报告 |
| Bug修复 → 检查 | 「修 bug」 | Skill-03 → Skill-02 | 修复代码 + 检查报告 |
| 安全修复 → 检查 | 「修安全漏洞」 | Skill-04 → Skill-02 | 修复代码 + 检查报告 |
| 复用 → 实现 → 检查 | 标准开发流程 | Skill-05 → Skill-01 → Skill-02 | 完整产出 |
