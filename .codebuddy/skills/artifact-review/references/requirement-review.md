# 需求分析报告审核标准

## 审核维度与权重

| 维度 | 权重 | 说明 |
|------|------|------|
| 需求完整性 | 25% | 功能边界、业务场景、用户故事是否完整 |
| 需求清晰度 | 20% | 描述是否无歧义、可理解、可执行 |
| 可行性分析 | 20% | 技术可行性、资源评估、工期合理性 |
| 风险评估 | 15% | 潜在风险识别与应对措施 |
| 验收标准 | 10% | 验收条件是否明确、可度量 |
| 文档规范 | 10% | 格式规范、结构清晰、引用准确 |

## 各维度详细检查项

### 需求完整性（25%）

- [ ] **功能范围**：明确定义了哪些功能在本次需求范围内，哪些不在
- [ ] **用户故事**：覆盖主要用户角色和使用场景
- [ ] **业务流程**：关键业务流程有完整描述（流程图或文字）
- [ ] **数据需求**：输入/输出数据格式、数据量级、数据来源有说明
- [ ] **非功能需求**：性能、安全、可用性等非功能需求有描述
- [ ] **边界条件**：异常情况、边界场景有考虑
- [ ] **依赖关系**：与其他系统/模块的依赖关系已说明

**扣分参考**：
- 缺少核心功能描述：-3~-5分
- 未覆盖主要用户场景：-2~-3分
- 缺少非功能需求：-1~-2分
- 边界条件考虑不足：-1~-2分

### 需求清晰度（20%）

- [ ] **无歧义性**：需求描述唯一、无多种解读可能
- [ ] **可理解性**：开发人员无需额外沟通即可理解
- [ ] **术语一致**：全文术语定义一致，有必要的术语表
- [ ] **优先级**：需求有明确的优先级标注（P0/P1/P2 或 Must/Should/Could）
- [ ] **量化指标**：性能要求等有具体数字，避免"快速"、"稳定"等模糊词

**扣分参考**：
- 存在歧义描述：-2~-4分/处
- 缺少量化指标：-1~-2分
- 优先级不明确：-1~-2分

### 可行性分析（20%）

- [ ] **技术可行性**：核心技术方案有初步验证或说明
- [ ] **资源评估**：开发人力、时间估算合理
- [ ] **工期计划**：里程碑节点明确，排期合理
- [ ] **外部依赖**：第三方依赖、接口联调等已纳入计划
- [ ] **替代方案**：对于高风险方案，有备选方案说明

**扣分参考**：
- 技术方案明显不可行：-5~-8分
- 工期估算明显不合理：-3~-5分
- 未考虑外部依赖：-2~-3分

### 风险评估（15%）

- [ ] **风险识别**：列出主要业务/技术/资源风险
- [ ] **风险评级**：对风险进行概率和影响度评估
- [ ] **应对措施**：每个主要风险有对应的缓解措施
- [ ] **回滚方案**：关键变更有回滚/降级方案

**扣分参考**：
- 无风险评估章节：-5~-7分
- 风险识别不全面：-2~-3分
- 无应对措施：-3~-4分

### 验收标准（10%）

- [ ] **验收条件明确**：每个功能点有对应的验收标准
- [ ] **可度量性**：验收标准可测试、可验证
- [ ] **完整覆盖**：主要功能均有验收标准
- [ ] **负向测试**：包含异常场景的验收标准

**扣分参考**：
- 无验收标准：-5~-7分
- 验收标准模糊：-2~-3分

### 文档规范（10%）

- [ ] **结构完整**：包含背景、目标、需求详情、验收标准等标准章节
- [ ] **格式规范**：标题层级合理，表格/图表清晰
- [ ] **版本管理**：有版本号和修改记录
- [ ] **引用准确**：相关文档、接口文档等引用链接有效

## 常见问题示例

### 🔴 严重问题
- 核心功能描述缺失，开发无法据此实现
- 存在明显技术不可行的方案
- 无任何验收标准

### 🟡 中等问题
- 部分场景描述不清晰，需要补充说明
- 性能指标缺少具体数字
- 风险评估不够全面

### 🟢 建议改进
- 增加流程图辅助说明
- 补充术语表
- 添加更多异常场景
