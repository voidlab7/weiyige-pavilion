# 代码实现审核标准

## 审核维度与权重

| 维度 | 权重 | 说明 |
|------|------|------|
| 代码质量 | 25% | 可读性、命名规范、代码结构 |
| 逻辑正确性 | 25% | 业务逻辑、边界条件、错误处理 |
| 安全性 | 20% | 安全漏洞、输入校验、权限控制 |
| 性能 | 15% | 时间复杂度、资源使用、数据库查询 |
| 可维护性 | 10% | 模块化、注释、测试友好性 |
| 规范遵从 | 5% | 编码规范、提交规范 |

## 各维度详细检查项

### 代码质量（25%）

**命名规范**
- [ ] 变量/函数/类命名有意义，见名知意
- [ ] 命名风格一致（camelCase/snake_case 等）
- [ ] 避免单字母变量（除循环变量外）
- [ ] 布尔变量命名以 is/has/can 开头

**代码结构**
- [ ] 函数长度适中（建议不超过 50 行）
- [ ] 单一职责：每个函数/类只做一件事
- [ ] 避免深层嵌套（建议不超过 3 层）
- [ ] 重复代码已提取为公共函数（DRY 原则）
- [ ] 魔法数字/字符串已提取为常量

**可读性**
- [ ] 复杂逻辑有注释说明
- [ ] 代码自解释，无需过多注释
- [ ] 注释与代码保持同步

**扣分参考**：
- 大量无意义命名（如 a、b、temp1）：-3~-5分
- 函数超过 100 行：-2~-3分/处
- 深层嵌套（超过 5 层）：-2~-3分/处
- 大量重复代码：-3~-5分

### 逻辑正确性（25%）

**业务逻辑**
- [ ] 核心业务逻辑实现正确
- [ ] 状态流转逻辑正确
- [ ] 计算逻辑无误（注意精度问题）

**边界条件**
- [ ] 空值/null 处理完整
- [ ] 数组越界保护
- [ ] 整数溢出处理（特别是金额计算）
- [ ] 并发场景下的竞态条件处理

**错误处理**
- [ ] 异常被适当捕获和处理（不吞异常）
- [ ] 错误信息有意义，便于排查
- [ ] 资源（文件、连接、锁）在异常时正确释放
- [ ] 重试逻辑有退避机制，避免雪崩

**扣分参考**：
- 业务逻辑明显错误：-5~-10分
- 空指针/越界未处理：-3~-5分/处
- 吞异常（catch 后不处理）：-3~-4分/处
- 资源泄露：-3~-5分/处

### 安全性（20%）

**输入校验**
- [ ] 所有外部输入（API参数、用户输入）有校验
- [ ] SQL 查询使用参数化查询，防止 SQL 注入
- [ ] 防止 XSS：输出到 HTML 时有转义
- [ ] 防止路径遍历：文件路径有校验

**认证鉴权**
- [ ] 接口有适当的认证机制
- [ ] 权限校验在服务端执行
- [ ] 敏感操作有二次确认或审计日志

**数据安全**
- [ ] 密码等敏感信息使用强加密（不使用 MD5）
- [ ] 日志中不打印敏感信息（密码、token、身份证等）
- [ ] 敏感数据传输使用 HTTPS
- [ ] Token/密钥不硬编码在代码中

**扣分参考**：
- SQL 注入漏洞：-8~-10分
- 密码明文存储：-8~-10分
- 硬编码密钥/密码：-6~-8分
- 无权限校验的敏感接口：-5~-8分
- 日志打印敏感信息：-3~-5分

### 性能（15%）

**算法效率**
- [ ] 核心算法时间复杂度合理
- [ ] 避免不必要的循环嵌套（O(n²) 以上需要说明）
- [ ] 大数据量处理使用分批/流式处理

**数据库**
- [ ] 查询有合适的索引
- [ ] 避免 N+1 查询问题
- [ ] 大查询有分页处理
- [ ] 事务范围最小化

**资源使用**
- [ ] 数据库连接使用连接池
- [ ] 大对象及时释放，避免内存泄漏
- [ ] 合理使用缓存，避免重复计算

**扣分参考**：
- N+1 查询：-3~-5分/处
- 全表扫描（无索引）：-3~-5分
- 内存泄漏：-3~-5分
- 事务范围过大：-2~-3分

### 可维护性（10%）

- [ ] **模块化**：功能模块划分合理，依赖关系清晰
- [ ] **可测试性**：核心逻辑可单独测试，依赖可 mock
- [ ] **配置外化**：环境相关配置（数据库地址、超时时间等）不硬编码
- [ ] **日志完整**：关键操作有日志记录，日志级别合理
- [ ] **向后兼容**：接口变更考虑了向后兼容性

**扣分参考**：
- 核心逻辑无法单元测试：-3~-4分
- 配置硬编码：-2~-3分
- 关键操作无日志：-2~-3分

### 规范遵从（5%）

- [ ] 遵守团队/语言编码规范
- [ ] 代码通过 lint 检查
- [ ] 提交信息规范（如遵循 Conventional Commits）
- [ ] 无调试代码残留（console.log、print 等）

## 常见问题示例

### 🔴 严重问题
```python
# ❌ SQL 注入漏洞
query = f"SELECT * FROM users WHERE name = '{user_input}'"

# ✅ 应使用参数化查询
query = "SELECT * FROM users WHERE name = %s"
cursor.execute(query, (user_input,))
```

```python
# ❌ 密码明文存储
db.save({"password": user_password})

# ✅ 应使用强哈希
db.save({"password": bcrypt.hash(user_password)})
```

### 🟡 中等问题
```python
# ❌ 吞异常
try:
    do_something()
except Exception:
    pass  # 问题被隐藏

# ✅ 应记录或处理
try:
    do_something()
except Exception as e:
    logger.error(f"操作失败: {e}")
    raise
```

```python
# ❌ N+1 查询
for user in users:
    orders = db.query(f"SELECT * FROM orders WHERE user_id = {user.id}")

# ✅ 批量查询
user_ids = [u.id for u in users]
orders = db.query("SELECT * FROM orders WHERE user_id IN (?)", user_ids)
```
