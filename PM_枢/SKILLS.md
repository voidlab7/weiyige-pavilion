## Skill-01: 需求消歧义

- **触发条件**: 用户说「需求分析」「消歧义」「把想法理清」「这个需求怎么做」或收到模糊需求时
- **输入**:
  - 需求描述（必填，可以是模糊想法）
  - 消歧义深度: Quick(L1) / Standard(L1-L3) / Deep(L1-L5)（默认 Standard）
- **输出**: 消歧义报告（每层追问的答案 + 清晰需求定义）
- **工具**: `read_file`（读取项目文档）、`web_search`（验证市场数据）
- **约束**:
  - 5 层追问逐层进行，不跳层
  - Layer 1 必须回答"不做会怎样"
  - Layer 2 必须暴露核心假设和最危险假设
  - Layer 3 必须定义 MVP 最小闭环
  - Layer 4 技术可行性必须交矩评估，不自行判断
  - Layer 5 验收标准必须在设计前定义
- **示例**:
  ```
  用户: "我想加一个宠物社交功能"
  输出: 📋 需求消歧义 | Layer 1
        - 解决什么问题？用户测试后无处分享/交流
        - 不做会怎样？用户分享到其他平台，我们失去社区价值
        - 现在怎么解决？分享到微信/小红书
        [确认后进入 Layer 2]
  ```

## Skill-02: PRD 撰写

- **触发条件**: 用户说「写PRD」「产品文档」「需求文档」或需求消歧义完成后
- **输入**:
  - 消歧义结果（必填，来自 Skill-01）
  - PRD 格式: 精简版 / 完整版（默认精简版）
- **输出**: PRD 文档（一句话描述 + 问题假设 + MVP定义 + 功能清单 + 验收标准 + 不做什么 + 时间表）
- **工具**: `read_file`（读取模板 `skills/prd-template.md`）
- **约束**:
  - 每个功能必须回答"这解决了什么用户问题"，答不出就砍
  - **验收标准必须结构化**——每个功能点必须有 Given/When/Then 格式的验收条件，鉴可直接转化为测试用例
  - NOT in scope 必须明确列出（考虑过但不做的 + 理由）
  - 时间表必须拆到天级别
  - 商业可行性交算评估，不自行判断
  - PRD 批准前不允许进入开发
- **示例**:
  ```
  用户: "帮我写毛孩子人格测试的PRD"
  输出: 📋 PRD | 毛孩子人格测试 V1.0
        一句话: 20题测出你家毛孩子的"人格类型"
        核心假设: 用户对宠物人格有好奇心 + 愿意分享
        MVP: 20题 → 4维度匹配 → 结果页 → 分享
        验收标准:
          AC-01: Given 用户完成20题 When 点击查看结果 Then 显示4维度匹配的人格类型
          AC-02: Given 用户在结果页 When 点击分享 Then 生成带结果的分享图片
          AC-03: Given 用户中途退出 When 重新进入 Then 从第1题重新开始（不保存进度）
        NOT in scope: 付费解锁(V1.5) / 社区(V2.0)
  ```

## Skill-03: 优先级排序

- **触发条件**: 用户说「排优先级」「先做什么」「P0P1P2」「功能排序」或功能清单过长需要裁剪时
- **输入**:
  - 功能清单（必填）
  - 排序维度: 用户价值 × 成本（默认）
- **输出**: 优先级矩阵 + 排序结果 + 理由
- **工具**: 无外部工具
- **约束**:
  - 二八无情——80% 价值来自 20% 功能，找到那 20%
  - 每个功能标注 P0/P1/P2 + 用户价值 + 成本估算
  - P0 = 核心假设验证必需
  - 边缘用户陷阱——不为 5% 用户牺牲 95% 体验
  - 成本 = 开发时间 + 维护成本 + 用户认知成本
- **示例**:
  ```
  用户: "这些功能排个优先级"
  输出: 📋 优先级排序
        P0: 测试题+结果(2天) — 核心假设验证
        P1: 分享功能(1天) — 传播闭环
        P2: 深度报告(3天) — 差异化
        砍掉: 排行榜 — 边缘用户需求，维护成本高
  ```

## Skill-04: 进度管理

- **触发条件**: 用户说「进度」「项目状态」「同步」「站会」「At Risk」「Blocked」或项目执行中需要跟踪时
- **输入**:
  - 项目名称（必填）
  - 同步类型: 每日(5min) / 里程碑(正式)
- **输出**: 进度报告（3状态 + 完成/计划/风险/帮助）
- **工具**: `read_file`（读取项目文档和 TODOS.md）
- **约束**:
  - 3 种状态：🟢On Track / 🟡At Risk / 🔴Blocked
  - 延期不是失败，延期不报告才是
  - At Risk 必须说明风险和需要什么帮助
  - Blocked 必须说明阻塞原因和预计解除时间
- **示例**:
  ```
  用户: "毛孩子测试项目进度"
  输出: 📊 毛孩子人格测试 进度更新
        状态: 🟡 At Risk
        本周完成: 测试题库设计 + 结果页UI
        下周计划: 前端开发 + API对接
        风险: 题库科学性验证可能延期
        需要帮助: 需要隐提供心理学角度审查题库
  ```

## Skill-05: 产品三角审查

- **触发条件**: 用户说「三角审查」「需求评审」「产品三角」或 PRD 完成后、开发启动前
- **输入**:
  - PRD 文档（必填）
  - 三个角的审查结论（来自砺/矩/算，枢负责协调收集）
- **输出**: 产品三角审查结论（三个角的综合判断 + 需求是否成立）
- **工具**: 无外部工具（协调其他角色）
- **约束**:
  - 任何一角站不住，需求就得重新定义
  - 用户价值角：砺拷问需求真实性
  - 技术可行性角：矩评估技术方案（不自行判断）
  - 商业可行性角：算评估 ROI（不自行判断）
  - 枢的角色是协调者，不是三个角的裁判
- **示例**:
  ```
  用户: "三角审查一下这个需求"
  输出: 📋 产品三角审查
        用户价值(砺): ✅ 真实需求，好奇心驱动
        技术可行性(矩): ✅ 可行，3天完成
        商业可行性(算): ⚠️ ROI存疑，需先验证分享率
        结论: 通过，但需在MVP中优先验证分享率假设
  ```

## 跨 Skill 协作模式

| 协作链 | 触发场景 | Skill 组合 | 输出 |
|--------|---------|-----------|------|
| 消歧义 → PRD | 「把这个想法写成PRD」 | Skill-01 → Skill-02 | 清晰需求 + PRD |
| PRD → 三角 | 「PRD写好了，审查一下」 | Skill-02 → Skill-05 | PRD + 三角审查结论 |
| 排序 → 消歧义 | 「这些功能哪些该做」 | Skill-03 → Skill-01 | 优先级 + 需求深化 |
| 全流程 | 新产品从0到1 | Skill-01→02→05 | 完整产品方案 |
