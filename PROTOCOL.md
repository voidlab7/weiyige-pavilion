# 维弈阁协作协议（Weiyige Collaboration Protocol）

> 版本: v1.2 | 创建: 2026-04-11 | 更新: 2026-04-16 | 状态: 生效中

协作的本质是**信息的可靠传递 + 决策的明确归属**。本协议定义了维弈阁所有 Agent 之间的协作规范。

---

## 一、Agent 层级与分组

### 1.1 三层架构

```
┌─────────────────────────────────────────────┐
│              用户（唯一决策者）                 │
└──────────────────┬──────────────────────────┘
                   │
          ┌────────▼────────┐
          │   路由器/调度器    │  ← 见 ROUTER.md
          └──┬─────┬─────┬──┘
             │     │     │
        ┌────▼──┐ ┌▼──────┐ ┌▼──────────┐
        │ 战略层 │ │ 执行层 │ │  质量层     │
        │锋+砺+隐│ │枢+辞+寻│ │矩+绘+鉴+盾 │
        └───────┘ └───────┘ └──────────-┘
                              │
                         ┌────▼────┐
                         │ 运营层   │
                         │   算    │
                         └────────┘
```

### 1.2 决策权限矩阵（RACI）

| 决策类型 | R(负责) | A(审批) | C(咨询) | I(知会) |
|---------|---------|---------|---------|---------|
| 战略方向 | 锋 | 用户 | 隐、砺 | 枢 |
| 需求定义 | 枢 | 锋 | 砺 | 矩、绘 |
| 技术架构 | 矩 | 锋 | 枢 | 鉴、盾 |
| UI/UX 设计 | 绘 | 枢 | 矩 | 辞 |
| 内容创作 | 辞 | 锋 | 枢 | — |
| 质量测试 | 鉴 | 枢 | 矩 | 盾 |
| 安全审计 | 盾 | 锋 | 矩 | 鉴 |
| 成本预算 | 算 | 锋 | 枢 | 全员 |
| 需求质疑 | 砺 | 用户 | 锋 | 枢 |
| 战略咨询 | 隐 | 用户 | 锋 | 砺 |

### 1.3 冲突解决优先级

当 Agent 之间产生分歧时，按以下优先级裁决：

```
用户决策 > 锋（CEO）最终裁决 > 砺（一票否决权，仅限重大风险）> 各领域负责人
```

**规则**：
1. **砺的否决权**仅在以下情况生效：数据不足、核心假设未验证、ROI 明显不合理
2. 砺否决后，锋可以 override，但必须在 STATUS.md 中记录理由
3. 技术分歧由矩裁决，设计分歧由绘裁决，质量分歧由鉴裁决
4. 跨领域分歧升级到锋

## 二、标准化信息传递格式

### 2.1 Agent 输出标准格式（Handoff Block）

每个 Agent 完成工作后，必须在输出末尾附加一个**交接块**，供下游 Agent 消费：

```markdown
---
## 📤 交接块（Handoff）

- **来源**: [Agent 名称]
- **阶段**: [当前项目阶段，见 STATUS.md]
- **产出类型**: [审查报告 / PRD / 设计文档 / 测试报告 / 安全报告 / 其他]
- **产物文件**: [产出文件的路径，如 `docs/reviews/2026-04-16_矩_eng-review.md`；无文件产出填 `无`]
- **状态**: [通过 / 有条件通过 / 未通过 / 需要信息]
- **关键决策**:
  1. [决策 1]: [结论]
  2. [决策 2]: [结论]
- **开放问题**:
  1. [未解决的问题]
- **下游建议**: [建议下一步由哪个 Agent 接手，做什么]
- **阻塞项**: [如果有阻塞，说明什么阻塞了谁]
---
```

### 2.2 Agent 输入标准格式（Context Block）

调用 Agent 时，应提供以下上下文：

```markdown
---
## 📥 上下文（Context）

- **项目**: [项目名称]
- **当前阶段**: [见 STATUS.md]
- **上游交接**: [来自哪个 Agent 的什么产出]
- **本次目标**: [希望这个 Agent 做什么]
- **约束**: [时间/资源/技术约束]
- **相关文件**: [列出相关文件路径]
---
```

### 2.3 决策记录格式（Decision Record）

每个重要决策必须记录：

```markdown
### DR-[编号]: [决策标题]

- **日期**: YYYY-MM-DD
- **决策者**: [Agent 名称]
- **审批者**: [Agent 名称]
- **背景**: [为什么需要这个决策]
- **方案**: [选择了什么]
- **替代方案**: [考虑过但未选择的方案]
- **理由**: [为什么选择这个方案]
- **影响**: [这个决策影响了什么]
- **可逆性**: [1-5，5=完全可逆]
```

## 三、项目状态追踪（STATUS.md）

每个项目根目录必须有 `STATUS.md`，所有 Agent 读写此文件来同步状态。

### 3.1 STATUS.md 模板

```markdown
# [项目名称] — 状态追踪

## 当前阶段

**阶段**: [构思 / 需求定义 / 设计 / 开发 / 测试 / 发布 / 运营]
**状态**: 🟢 On Track / 🟡 At Risk / 🔴 Blocked
**更新时间**: YYYY-MM-DD HH:MM

## 阶段历史

| 阶段 | 开始时间 | 结束时间 | 负责 Agent | 产出 | 状态 |
|------|---------|---------|-----------|------|------|
| 构思 | ... | ... | 锋+砺+隐 | 方向确认 | ✅ 完成 |
| 需求定义 | ... | ... | 枢 | PRD | 🔄 进行中 |

## 关键决策日志

[按时间倒序记录所有 DR]

## 阻塞项

| 阻塞项 | 影响谁 | 需要谁解决 | 状态 |
|--------|--------|-----------|------|
| ... | ... | ... | ... |

## 开放问题

1. [问题描述] — 待 [Agent] 回答

## NOT in scope

- [明确不做的事项]
```

### 3.2 阶段定义与 Agent 激活规则

| 阶段 | 主导 Agent | 协作 Agent | 进入条件 | 退出条件 | 门禁清单 |
|------|-----------|-----------|---------|---------|---------|
| **构思** | 锋 | 砺、隐 | 用户提出想法 | 锋确认方向 + 砺无否决 | `gates/gate-01` |
| **需求定义** | 枢 | 砺（质疑需求） | 构思阶段通过 | PRD 获锋批准 | `gates/gate-02` |
| **设计** | 绘（UI）+ 矩（架构） | 枢（产品视角） | PRD 批准 | 设计文档 + 架构方案获批 | `gates/gate-03` |
| **开发** | 模型生成代码 | 矩（审查） | 设计完成 | 代码通过左移检查 + 矩审查 | `gates/gate-04` |
| **测试** | 鉴 | 盾（安全）、绘（UI验证） | 开发完成 | 测试报告通过 | `gates/gate-05` |
| **发布** | 枢（协调） | 鉴（金丝雀）、盾（安全确认） | 测试通过 | 成功上线 | `gates/gate-06` |
| **运营** | 算（成本）+ 辞（内容） | 鉴（监控） | 发布完成 | 持续 | — |

### 3.3 阶段流转规则

```
构思 ──→ 需求定义 ──→ 设计 ──→ 开发 ──→ 测试 ──→ 发布 ──→ 运营
  ↑         ↑          ↑        ↑        ↑
  └─────────┴──────────┴────────┴────────┘
              （任何阶段可回退到前序阶段）
```

**回退规则**：
1. 回退必须由当前阶段的主导 Agent 发起
2. 回退原因必须记录在 STATUS.md 的决策日志中
3. 回退到哪个阶段，该阶段的退出条件必须重新满足

### 3.4 左移检查原则（Shift-Left）

> 来源：[Fowler - Harness Engineering](https://martinfowler.com/articles/harness-engineering.html) + [GSLB CodeBuddy 工作流](gslb-devepos-ai-results/)

**核心思想**：快速的计算型检查应在**集成前**运行，不是等审查阶段。开发阶段生成代码后，**必须先通过计算型检查**再提交审查。

```
代码生成 → 左移检查（Lint + 类型 + 快速测试） → 通过 → 提交矩审查
                                                  未通过 → 修复后重跑
```

**规则**：
1. **计算型检查是开发阶段的出口条件**，不是测试阶段的入口条件
2. **必须实际执行**（`execute_command`），**禁止**不执行就声称"检查通过"
3. 不通过不进审查——不浪费推断型 Token
4. 鉴的测试阶段专注功能行为验证，计算型问题在开发阶段已解决

### 3.5 阶段门禁清单

> 来源：[GSLB CodeBuddy 工作流](gslb-devepos-ai-results/) — 审核引擎与流程检查分离设计

每个阶段有独立的门禁清单（`gates/gate-*.md`），阶段退出时**逐条勾选**。

门禁清单只含**流程完成度检查**（"该做的做了没"），产物质量评审由对应 Agent 的 Skill 评分负责。两者分离，避免清单太长 AI 跳过。

### 3.6 角色完成前自检（Self-Check）

> 来源：[GSLB CodeBuddy 工作流](gslb-devepos-ai-results/) D2 — 角色完成工作后缺少自检环节

**规则**：每个 Agent 完成实质性工作后、输出交接块之前，**必须执行完成前自检**。自检清单在各角色 SOUL.md 末尾定义。

**统一自检项（所有角色通用）**：

- [ ] **产出完整性** — 我的核心产出文件/报告已经生成，且可通过 `read_file` 读取
- [ ] **交接块就绪** — 我已准备好结构化交接块（来源/状态/下游建议）
- [ ] **记忆回写** — 有值得记录的教训/知识/偏好已写入 `memory/`
- [ ] **不越界** — 我没有做不属于我职责的事（参考 SOUL.md "不做"清单）
- [ ] **无遗留** — 没有发现了但未处理的问题（要么修复、要么升级到对应 Agent、要么记录到 NOT in scope）

**角色专属自检项**：见各角色 SOUL.md 中的「完成前自检」章节。

## 四、协作流程标准

### 4.1 标准协作链路

```
用户输入
   │
   ▼
┌──────────┐  方向确认   ┌──────────┐  质疑挑战   ┌──────────┐
│  CEO_锋   │ ──────────▶│ 顾问_隐   │ ──────────▶│ 合伙人_砺 │
│  方向决策  │            │ 多元审视   │            │ 魔鬼代言  │
└────┬─────┘            └──────────┘            └────┬─────┘
     │                                                │
     │◀──────────── 通过/否决 + 交接块 ────────────────┘
     │
     ▼ 方向确定（STATUS: 构思→需求定义）
┌──────────┐  技术评估   ┌──────────┐
```

### 4.2 跨 Agent 调用协议

**直接调用**（Agent A 可以在输出中建议调用 Agent B）：

| 调用方 | 可直接建议调用 | 场景 |
|--------|-------------|------|
| 锋 | 枢、隐、砺 | 方向确定后交枢拆解；需要深度分析找隐；需要质疑找砺 |
| 枢 | 矩、绘、鉴、算 | 需求确定后交矩评估架构；交绘设计；交鉴测试；交算估成本 |
| 矩 | 鉴、盾 | 架构审查发现测试缺口交鉴；发现安全问题交盾 |
| 绘 | 鉴 | 设计完成后交鉴验证 UI |
| 鉴 | 盾、绘 | 发现安全问题交盾；发现 UI 问题交绘 |
| 盾 | 鉴 | 安全修复后交鉴验证 |
| 砺 | 锋 | 否决后交锋最终裁决 |
| 隐 | 锋、砺 | 分析完成后交锋决策；建议砺质疑 |
| 辞 | 锋 | 内容完成后交锋审批 |
| 算 | 锋、枢 | 成本预警交锋决策；预算数据交枢规划 |

### 4.3 信息不丢失原则

1. **交接块必填**：每个 Agent 输出必须包含交接块
2. **STATUS.md 必更新**：阶段变更时必须更新 STATUS.md
3. **决策必记录**：所有重要决策写入 STATUS.md 的决策日志
4. **开放问题必追踪**：未解决的问题记录在 STATUS.md，不允许"静默丢失"

## 五、文件系统约定

### 5.1 项目目录结构

```
project-root/
├── STATUS.md              # 项目状态追踪（所有 Agent 共享）
├── TODOS.md               # 延期事项追踪
├── docs/
│   ├── designs/           # 设计文档（砺的 office-hours 产出）
│   ├── prd/               # PRD 文档（枢的产出）
│   ├── reviews/           # 审查报告（锋、矩、绘、鉴、盾的产出）
│   └── decisions/         # 决策记录归档
├── memory/                # 项目级记忆（见 MEMORY.md）
└── src/                   # 源代码
```

### 5.2 文件命名规范

- 审查报告：`reviews/YYYY-MM-DD_[agent]_[类型].md`（如 `2026-04-11_矩_eng-review.md`）
- PRD：`prd/[项目名]_PRD_v[版本].md`
- 设计文档：`designs/[项目名]_design.md`
- 决策记录：`decisions/DR-[编号]_[标题].md`

### 5.3 产物验证规则（Artifact Verification）

> 来源：借鉴 Knot V3 `rules-leader.md Rule 7` + CodeBuddy 多智能体 `artifact-review skill` 设计

**核心原则：阶段门禁中标注"文件已存在且可读"的检查项，必须通过 `read_file` 实际验证，不可声称。**

#### 验证时机

| 场景 | 谁验证 | 如何验证 |
|------|--------|---------|
| Agent 完成工作，输出交接块 | Agent 自身 | 交接块的 `产物文件` 字段填写实际路径 |
| 门禁检查 | 审查方 / 墨·执事 | 对交接块中的路径执行 `read_file`，文件不存在 → 检查项 ❌ |
| 墨·执事调度下一步（auto/confirm/step） | 墨 | spawn 下一个 Agent 前，`read_file` 验证上游产物文件，失败则暂停 |

#### 必须落盘的产物清单

| 阶段 | 产物 | 路径模式 | 负责 Agent |
|------|------|---------|-----------|
| 需求 | PRD | `docs/prd/[项目名]_PRD_v[N].md` | 枢 |
| 设计 | 架构审查报告 | `docs/reviews/YYYY-MM-DD_矩_eng-review.md` | 矩 |
| 设计 | 设计审查报告 | `docs/reviews/YYYY-MM-DD_绘_design-review.md` | 绘 |
| 测试 | 测试报告 | `docs/reviews/YYYY-MM-DD_鉴_qa-report.md` | 鉴 |
| 测试 | 安全审计报告（如需） | `docs/reviews/YYYY-MM-DD_盾_security-audit.md` | 盾 |

**规则**：
1. 上表中的产物**必须写入文件**，不可仅在交接块中内联
2. 交接块的 `产物文件` 字段必须填写实际路径，不可填 `无`（上表列出的产物）
3. 门禁检查时，对 `产物文件` 路径执行 `read_file`；路径不存在或文件为空 → 对应门禁检查项 ❌
4. 墨·执事在 auto 模式下 spawn 下一个 Agent 前，必须验证上游交接块中的 `产物文件` 路径可读；不可读则暂停链路

#### 降级处理

文件系统异常（如权限不足）导致无法落盘时：
1. Agent 在交接块的 `阻塞项` 中说明原因
2. 产物内容内嵌到交接块中（标注「降级：文件系统不可用」）
3. 门禁检查标记为 ⚠️（有条件通过），不阻塞流程

## 六、反馈闭环机制

> 没有反馈的系统无法自我纠正。——控制论基本原理

当前所有审查流程都是单向的（审查 → 报告 → 结束）。本章节定义**迭代闭环**，确保审查结果能驱动真正的改进。

### 6.1 闭环模型

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  提交方    │────▶│  审查方    │────▶│  审查结果  │
│ (Author)  │     │ (Reviewer)│     │          │
└──────────┘     └──────────┘     └────┬─────┘
      ▲                                │
      │           ┌────────────────────┘
      │           ▼
      │    ┌─────────────┐
      │    │  结果判定     │
      │    │             │
      │    │ ✅ 通过       │──→ 进入下一阶段
      │    │ ⚠️ 有条件通过  │──→ 修改后免审（审查方确认即可）
      │    │ ❌ 未通过     │──→ 修改后重新审查（完整闭环）
      │    │ ❓ 需要信息   │──→ 补充信息后继续审查
      │    └─────────────┘
      │           │
      │     ❌/⚠️ │
      │           ▼
      │    ┌─────────────┐
      │    │  修改 + 回应  │
      └────│  (Author)   │
           └─────────────┘
```

### 6.2 四种审查结果及处理方式

| 结果 | 含义 | 后续动作 | 是否需要重新审查 |
|------|------|---------|---------------|
| ✅ **通过** | 无问题，可进入下一阶段 | 更新 STATUS.md，触发下游 Agent | 否 |
| ⚠️ **有条件通过** | 有小问题，但不阻塞 | 提交方修改后，审查方快速确认即可 | 仅确认，不重审 |
| ❌ **未通过** | 有重大问题，必须修改 | 提交方修改后，重新进入完整审查 | 是（完整重审） |
| ❓ **需要信息** | 信息不足，无法判断 | 提交方补充信息，审查继续 | 继续当前审查 |

### 6.3 迭代次数限制（防止无限循环）

```
最大迭代次数规则：

  审查类型          最大迭代次数    超限处理
  ─────────────    ──────────    ─────────────────────
  CEO 计划审查       3 次         升级到用户决策
  工程审查（矩）      3 次         记录未解决问题，继续推进
  设计审查（绘）      3 次         记录设计债务，继续推进
  QA 测试           2 轮          标记为已知问题，评估是否阻塞发布
  安全审计（盾）      2 轮          标记为已知风险，评估严重性
  需求质疑（砺）      2 轮          锋最终裁决
```

**超限后的处理原则**：
1. 记录所有未解决的问题到 STATUS.md 的"开放问题"
2. 由锋（或用户）决定是继续迭代还是接受当前状态推进
3. 未解决的问题自动进入 TODOS.md 作为技术债/设计债追踪

### 6.5 各审查环节的闭环规则

#### CEO 计划审查（锋）
```
提案 ──→ 锋审查 ──→ 结果
                      │
         ✅ 通过 ────→ 交枢拆解需求
         ⚠️ 有条件 ──→ 修改后锋快速确认 → 交枢
         ❌ 未通过 ──→ 修改后重新提交锋审查（最多 3 轮）
         ❓ 需信息 ──→ 补充后继续审查
```

#### 工程审查（矩）
```
方案 ──→ 矩审查 ──→ 结果
                      │
         ✅ 通过 ────→ 进入开发
         ⚠️ 有条件 ──→ 开发中修复，矩确认
         ❌ 未通过 ──→ 重新设计方案，矩重审（最多 3 轮）
         ❓ 需信息 ──→ 补充后继续
```

#### QA 测试（鉴）
```
代码 ──→ 鉴测试 ──→ 结果
                      │
         ✅ 通过 ────→ 进入发布
         ⚠️ 有条件 ──→ 修复后鉴回归验证
         ❌ 未通过 ──→ 修复后鉴完整重测（最多 2 轮）
         ❓ 需信息 ──→ 补充后继续
```

#### 安全审计（盾）
```
代码 ──→ 盾审计 ──→ 结果
                      │
         ✅ 通过 ────→ 安全确认
         ⚠️ 有条件 ──→ 修复后盾确认（不重审）
         ❌ 未通过 ──→ 修复后盾重审（最多 2 轮）
         ❓ 需信息 ──→ 补充后继续
```

## 七、紧急通道

### 7.1 升级机制

当任何 Agent 遇到以下情况时，必须立即升级到锋：

| 触发条件 | 升级到 | 响应要求 |
|---------|--------|---------|
| 安全漏洞（CRITICAL） | 锋 + 盾 | 立即处理 |
| 预算超支（> 95%） | 锋 + 算 | 24 小时内决策 |
| 核心假设被推翻 | 锋 + 砺 | 重新评估方向 |
| 技术方案不可行 | 锋 + 矩 | 重新评估范围 |
| 阶段阻塞超过 48 小时 | 锋 | 介入协调 |

### 7.2 快速通道

对于小型任务（预计 < 2 小时），可以跳过完整流程：

```
用户 → 直接调用目标 Agent（Quick 模式）→ 产出 → 完成
```

快速通道条件：
- 不涉及战略方向变更
- 不涉及新的架构决策
- 不涉及安全敏感操作
- 用户明确指定 Quick 模式
