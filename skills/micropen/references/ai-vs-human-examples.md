# ai-vs-human-examples.md — AI 初稿 vs 人工修改对比

> v2.0 新增。让 AI 看到自己的问题在哪，学会自我矫正。
> 每对范例都标注了「差在哪」和「改了什么」。

---

## 1. 假设性例子 → 正在发生的真实细节

**AI 写的**：
```
比如有一次我需要调试一个多 Agent 协作的问题，以前可能得手动一个个看日志。
```

**人改的**：
```
就像我今天下午正在调的维弈阁那个 bug，启·执事把任务派给了铸·开发，
但铸·开发执行到一半卡住了，日志里一片空白。
```

**差在哪**：「有一次」是编造的时间锚点，「今天下午」是正在发生的真实细节。AI 倾向于构造一个安全的假设场景，人会直接说正在干的事。

---

## 2. 知识性描述 → 体感记忆

**AI 写的**：
```
这个发现让我感到非常震撼，原来多 Agent 协作最大的瓶颈不在模型能力。
```

**人改的**：
```
我当时就愣住了。
不是模型不行，是根本没人管「链路」这回事。
```

**差在哪**：「感到非常震撼」是在描述一种情绪（像在写报告），「当时就愣住了」是在回忆一种体感（像在跟朋友聊）。

---

## 3. 空泛工具名 → 具体名称

**AI 写的**：
```
我尝试用一些 AI 编程工具来辅助开发，发现效率确实有提升。
```

**人改的**：
```
我拿 Claude Code 写了一天 Chromium 插件，中间让 CodeBuddy 帮忙 review，
发现它居然能捡出我漏掉的 out-of-line 构造函数。
```

**差在哪**：「一些 AI 编程工具」说了等于没说。具体到 Claude Code + CodeBuddy + 具体场景（Chromium 插件 + out-of-line 构造函数），可信度完全不同。

---

## 4. 教科书式科普 → 随手掏

**AI 写的**：
```
这里需要先介绍一个概念。在软件工程中，有一种设计模式叫「编排器模式」（Orchestrator Pattern），
它的核心思想是将任务分配和执行分离，由一个中心节点负责协调。
```

**人改的**：
```
突然想起以前在 Chromium 里见过一种类似的设计，叫 Orchestrator Pattern。
简单说就是有个角色专门负责调度，自己不干活。
启·执事就是干这事的。
```

**差在哪**：第一种像在读 Wikipedia，第二种像在跟朋友聊天时突然想起了一个东西。「突然想起」「简单说就是」「就是干这事的」——这些口语化的连接词是关键。

---

## 5. 过度工整 → 故意打破

**AI 写的**：
```
多 Agent 系统面临三个核心挑战：
1. 上下文管理：每个 Agent 的上下文如何传递
2. 任务编排：谁来决定执行顺序
3. 错误处理：某个 Agent 失败后如何恢复
```

**人改的**：
```
多 Agent 系统，坑在哪？

第一个，上下文。Agent A 干完了，怎么把它知道的东西传给 Agent B？
这个听起来简单，做起来要命。

第二个，编排。谁来管谁先干谁后干？没人管的话，13 个 Agent 就是 13 只无头苍蝇。

第三个。。。说实话第三个是我踩了三天才发现的。
错误恢复。铸·开发写崩了，启·执事得知道怎么重试，不是把整个链路推倒重来。
```

**差在哪**：AI 的列表工整得像 PPT，人的版本用疑问句开头、用比喻（无头苍蝇）、用省略号制造悬念、用「说实话」「踩了三天」加入个人经历。同样的信息，但第二种有温度。

---

## 6. 降落式结尾 → 弹射式结尾

**AI 写的**：
```
总的来说，多 Agent 协作是一个复杂但有价值的方向。希望这篇文章能给大家一些启发，
如果有任何问题欢迎在评论区交流。我们下期再见。
```

**人改的**：
```
造更多 Agent 不是答案，造更好的链路才是。

你的多 Agent 系统，跑通链路了吗？
```

**差在哪**：AI 的结尾软绵绵地降落了（「总的来说」「希望」「欢迎」「下期再见」全是禁用句式），人的结尾是弹射——一个金句 + 一个反问，读完脑子里会留下这句话。

---

## 7. 数字不够极端 → 拉到极端

**AI 写的**：
```
一个善用 AI 的人，每天工作六个小时的产出可能比不用 AI 的人工作十个小时还多。
```

**人改的**：
```
一个会用 AI 的人，每天干 2 个小时的活，可能比不用 AI 的人干 10 个小时还多。
```

**差在哪**：6 vs 10 的对比不够刺激，2 vs 10 才有冲击力。而且「工作六个小时的产出」太书面，「干 2 个小时的活」更口语。阿拉伯数字比汉字数字更有视觉冲击。

---

## 8. 平铺直叙 → 升番排列

**AI 写的**：
```
我测试了 Claude Code、CodeBuddy 和 Copilot，它们在多文件编辑方面各有优劣。
Claude Code 的上下文管理最好，CodeBuddy 的中文支持最强，Copilot 在补全方面领先。
```

**人改的**：
```
先试了 Copilot。嗯，补全很快，但写超过 50 行就开始抽风。及格。

再试 CodeBuddy。中文 prompt 直接上，不用翻译。而且它居然能读懂维弈阁的 AGENTS.md。有点意思。

最后上 Claude Code。
我跟它说了一句「帮我重构启·执事的调度逻辑」，它直接把 5 个文件改了。
改完我 review 了一下。

说实话，比我自己写的还清楚。
```

**差在哪**：AI 的写法是并列罗列（无感），人的写法是逐一展示 + 升番——从「及格」到「有点意思」到「说实话比我写的还清楚」，情绪层层递进。

---

## 使用方式

写完初稿后，把可疑段落跟上面这些范例对照：
1. 有没有假设性例子？→ 换成正在发生的真实细节
2. 有没有知识性描述？→ 换成体感记忆
3. 有没有空泛工具名？→ 换成具体名称 + 场景
4. 有没有教科书科普？→ 换成随手掏
5. 有没有过度工整？→ 故意打破
6. 结尾是降落还是弹射？→ 换成金句/反问
7. 数字有冲击力吗？→ 拉到极端
8. 多个案例是平铺还是升番？→ 弱→强排列

---

**AI 最大的问题不是写得「不对」，是写得「太对了」。太工整、太完备、太平衡——这些恰恰是暴露 AI 的特征。人写东西是有偏好、有遗漏、有口癖的。**
