# 测试实现审核标准

## 审核维度与权重

| 维度 | 权重 | 说明 |
|------|------|------|
| 测试覆盖率 | 25% | 功能覆盖、代码覆盖、场景覆盖 |
| 用例设计质量 | 25% | 用例的有效性、独立性、可维护性 |
| 边界与异常测试 | 20% | 边界值、异常流程、负向测试 |
| 测试可靠性 | 15% | 稳定性、可重复性、环境隔离 |
| 测试文档 | 10% | 用例描述、预期结果、前置条件 |
| 规范遵从 | 5% | 命名规范、测试框架使用规范 |

## 各维度详细检查项

### 测试覆盖率（25%）

**功能覆盖**
- [ ] 所有需求功能点都有对应测试用例
- [ ] 主要业务流程（Happy Path）全覆盖
- [ ] 核心 API 接口全覆盖
- [ ] 关键业务规则有专项测试

**代码覆盖**
- [ ] 行覆盖率达标（建议 ≥ 80%）
- [ ] 分支覆盖率达标（建议 ≥ 70%）
- [ ] 核心模块覆盖率更高（建议 ≥ 90%）

**场景覆盖**
- [ ] 正常流程（Happy Path）
- [ ] 异常流程（Error Path）
- [ ] 边界条件
- [ ] 并发场景（如适用）

**扣分参考**：
- 核心功能无测试：-5~-8分
- 覆盖率低于 60%：-4~-6分
- 覆盖率 60%-80%：-2~-3分
- 重要分支未覆盖：-3~-4分

### 用例设计质量（25%）

**有效性**
- [ ] 每个测试用例验证一个明确的行为
- [ ] 断言（Assert）有意义，不是永真断言
- [ ] 测试验证了正确的事情（不只是"不报错"）
- [ ] 测试数据有代表性

**独立性**
- [ ] 测试用例相互独立，无执行顺序依赖
- [ ] 每个测试自行准备数据，不依赖其他测试的副作用
- [ ] 测试后清理数据，不影响其他测试

**可维护性**
- [ ] 测试代码简洁，易于理解
- [ ] 使用 setUp/tearDown 或 fixture 管理测试数据
- [ ] 避免重复的测试代码（使用参数化测试）
- [ ] 测试辅助方法有合理抽象

**扣分参考**：
- 测试用例有执行顺序依赖：-4~-6分
- 断言为永真（如 `assert True`）：-3~-5分/处
- 大量重复测试代码：-2~-3分
- 测试数据不清理：-2~-3分

### 边界与异常测试（20%）

**边界值测试**
- [ ] 数值边界：最大值、最小值、零值、负值
- [ ] 字符串边界：空字符串、超长字符串、特殊字符
- [ ] 集合边界：空集合、单元素、最大元素数
- [ ] 时间边界：过去/未来时间、跨时区、闰年等

**异常与负向测试**
- [ ] 无效输入的错误处理
- [ ] 权限不足的拒绝场景
- [ ] 资源不存在的 404 场景
- [ ] 并发冲突场景
- [ ] 网络超时/服务不可用场景（如适用）

**扣分参考**：
- 无任何边界值测试：-5~-7分
- 无负向/异常测试：-5~-7分
- 边界值覆盖不全：-2~-3分

### 测试可靠性（15%）

**稳定性**
- [ ] 测试可以重复执行，结果一致（无 flaky test）
- [ ] 不依赖时间（如不依赖当前时间，而是 mock）
- [ ] 不依赖执行环境的特定状态

**隔离性**
- [ ] 单元测试不依赖外部服务（数据库、网络等已 mock）
- [ ] 测试数据与生产数据隔离
- [ ] 测试之间相互隔离，无全局状态污染

**速度**
- [ ] 单元测试执行速度快（单个用例 < 1 秒）
- [ ] 集成测试有合理的超时设置
- [ ] 慢速测试有标记，可选择性跳过

**扣分参考**：
- Flaky test（随机失败）：-4~-6分/处
- 单元测试依赖真实数据库：-3~-5分
- 测试执行极慢（>10分钟）：-2~-4分

### 测试文档（10%）

- [ ] **用例名称**：清晰描述测试场景（如 `test_login_with_invalid_password_should_return_401`）
- [ ] **前置条件**：说明测试所需的初始状态
- [ ] **测试步骤**：对于集成测试，步骤清晰
- [ ] **预期结果**：明确说明期望的输出/状态
- [ ] **测试数据**：关键测试数据有说明

**扣分参考**：
- 用例命名无意义（如 `test1`、`test_func`）：-2~-4分
- 无预期结果说明：-2~-3分

### 规范遵从（5%）

- [ ] 遵守测试框架的最佳实践（如 AAA 模式：Arrange-Act-Assert）
- [ ] 测试文件/类/方法命名规范
- [ ] 使用合适的断言方法（不滥用 assertTrue）
- [ ] 测试代码通过 lint 检查

## 常见问题示例

### 🔴 严重问题
```python
# ❌ 无意义断言（永真）
def test_create_user():
    result = create_user("alice", "password")
    assert result is not None  # 几乎不可能为 None

# ✅ 验证具体行为
def test_create_user():
    result = create_user("alice", "password123")
    assert result.username == "alice"
    assert result.id is not None
    assert result.created_at is not None
```

```python
# ❌ 测试有执行顺序依赖
def test_step1_create_order():
    global order_id
    order_id = create_order()  # 依赖全局变量

def test_step2_pay_order():
    pay_order(order_id)  # 依赖上一个测试
```

### 🟡 中等问题
```python
# ❌ 无边界值测试
def test_calculate_discount():
    assert calculate_discount(100) == 10  # 只测试了一个正常值

# ✅ 应包含边界值
@pytest.mark.parametrize("amount,expected", [
    (0, 0),        # 零值
    (1, 0),        # 最小值
    (99, 0),       # 边界下
    (100, 10),     # 边界值
    (1000, 100),   # 正常值
    (-1, ValueError),  # 负值（异常）
])
def test_calculate_discount(amount, expected):
    ...
```

### 🟢 建议改进
```python
# 建议：使用参数化测试减少重复
# 建议：使用 fixture 管理测试数据
# 建议：为 flaky test 添加重试机制并记录原因
```

## 测试类型说明

| 测试类型 | 覆盖目标 | 执行速度 | 隔离要求 |
|---------|---------|---------|---------|
| 单元测试 | 单个函数/方法 | 极快(<1s) | 完全隔离（mock外部依赖） |
| 集成测试 | 模块间交互 | 中等(1-30s) | 隔离外部服务 |
| 端到端测试 | 完整业务流程 | 慢(>30s) | 使用测试环境 |
| 性能测试 | 响应时间/吞吐量 | 视场景 | 独立测试环境 |
