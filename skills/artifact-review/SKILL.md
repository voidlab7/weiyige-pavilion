# 通用审核技能

> 维弈阁适配版

## 支持的审核类型

| 审核类型 | 触发关键词 | 参考文档 |
|---------|-----------|---------|
| 需求分析报告 | 需求文档、PRD、需求分析 | [references/requirement-review.md](references/requirement-review.md) |
| 技术方案文档 | 技术方案、设计文档、架构设计 | [references/tech-design-review.md](references/tech-design-review.md) |
| 代码实现 | 代码审查、Code Review、代码质量 | [references/code-review.md](references/code-review.md) |
| 测试实现 | 测试用例、测试报告、测试方案 | [references/test-review.md](references/test-review.md) |

## 工作流程

### 第一步：识别审核类型

根据用户提供的内容自动判断审核类型。若无法判断，询问用户。

### 第二步：加载对应参考文档

根据审核类型，读取对应的参考文档获取详细的审核维度和评分标准。

### 第三步：执行审核

1. 通读全部内容，理解整体结构和目标
2. 按各维度逐项分析，记录发现的问题
3. 区分问题严重程度：🔴 严重 / 🟡 中等 / 🟢 建议

### 第四步：多维度打分

按参考文档中定义的维度进行打分（每维度 0-10 分），计算加权总分（满分 100 分）。

### 第五步：输出审核报告

使用以下标准格式输出审核报告：

```markdown
## 📋 审核报告 - [文档/代码名称]

**审核类型**：[类型]  
**审核时间**：[时间]  
**总体评分**：[X/100] [等级]

---

### 📊 多维度评分

| 维度 | 权重 | 得分 | 加权分 | 评价 |
|------|------|------|--------|------|
| 维度1 | XX% | X/10 | X.X | 简评 |
| ... | ... | ... | ... | ... |
| **综合得分** | **100%** | - | **XX.X** | |

---

### 🔍 详细审核意见

#### [维度1]
- 🟢 亮点：...
- 🟡 问题：...
- 🔴 严重问题：...

...

---

### 💡 改进建议

**优先修复（P0）**：
1. ...

**建议优化（P1）**：
1. ...

**可选改进（P2）**：
1. ...

---

### ✅ 总结

[2-3句总体评价，包含主要优点和核心改进方向]
```

## 评分等级

| 总分 | 等级 | 说明 |
|------|------|------|
| 90-100 | ⭐⭐⭐⭐⭐ 优秀 | 质量极高，可直接使用 |
| 80-89 | ⭐⭐⭐⭐ 良好 | 质量较高，少量优化即可 |
| 70-79 | ⭐⭐⭐ 中等 | 基本合格，需要改进 |
| 60-69 | ⭐⭐ 及格 | 存在明显缺陷，需要较多修改 |
| 0-59 | ⭐ 不及格 | 质量较差，需要重大修改 |

## 与维弈阁的集成

- **锋·CEO**：使用本 Skill 做战略审查时，结合 `rules/review-scoring.md` 的 CEO 专属维度（方向契合度、ROI、资源效率）
- **矩·架构**：使用本 Skill 做工程审查时，结合 `rules/review-scoring.md` 的架构师专属维度（数据流、边界情况、可测试性）
- **绘·设计**：使用本 Skill 做 UI 审查时，结合 `rules/review-scoring.md` 的设计专属维度（视觉一致性、交互流畅度、无障碍）
- **鉴·QA**：使用本 Skill 做测试审查时，结合 `rules/review-scoring.md` 的 QA 专属维度（覆盖率、回归风险）
- **盾·安全**：使用本 Skill 做安全审查时，结合 `rules/review-scoring.md` 的安全专属维度（OWASP、注入、认证）

> 本 Skill 提供通用审核框架和打分模板，维弈阁 `rules/review-scoring.md` 提供角色专属维度。两者互补，不冲突。

## 注意事项

- 审核意见应具体、可操作，避免模糊描述
- 对于代码审核，指出具体的行号或代码片段
- 对于文档审核，指出具体的章节或段落
- 同时指出优点，不只是问题
- 若内容较长，优先关注高权重维度
