# 维弈阁 Memory 系统规范

> 版本: v1.1 | 创建: 2026-04-11 | 状态: 生效中

没有记忆的 Agent 只是一次性指令执行器。Memory 系统让 Agent 能从经验中学习，产生复利效应。

---

## 一、Memory 分层架构

```
┌─────────────────────────────────────────────┐
│           长期记忆（Long-term）                │
│  团队级知识库，跨项目持久化                      │
│  存储位置：各 Agent 的 memory/ 目录              │
├─────────────────────────────────────────────┤
│           中期记忆（Mid-term）                  │
│  项目级上下文，项目生命周期内持久化               │
│  存储位置：项目根目录 memory/ 目录               │
├─────────────────────────────────────────────┤
│           短期记忆（Short-term）                │
│  当前会话上下文，会话结束后沉淀关键信息           │
│  存储位置：对话上下文（自动）                     │
└─────────────────────────────────────────────┘
```

### 1.1 三层记忆说明

| 层级 | 生命周期 | 存储位置 | 写入时机 | 典型内容 |
|------|---------|---------|---------|---------|
| **长期** | 永久（手动归档） | `[Agent]/memory/` | 项目复盘、重要教训 | 团队偏好、通用教训、最佳实践 |
| **中期** | 项目周期 | 项目 `memory/` | 每个阶段结束时 | 项目决策、架构选择、需求变更 |
| **短期** | 单次会话 | 对话上下文 | 自动 | 当前讨论的细节 |

## 二、Memory 数据结构

### 2.1 长期记忆（Agent 级）

存储在各 Agent 的 `memory/` 目录下，按类型分文件：

```
[Agent]/memory/
├── preferences.md      # 用户偏好和工作习惯
├── lessons.md          # 经验教训（踩过的坑）
└── knowledge.md        # 积累的领域知识
```

#### preferences.md 格式

```markdown
# [Agent 名称] — 用户偏好记忆

> 最后更新: YYYY-MM-DD

## 沟通偏好

- [偏好 1]: [描述]
- [偏好 2]: [描述]

## 工作习惯

- [习惯 1]: [描述]
- [习惯 2]: [描述]

## 技术偏好

- [偏好 1]: [描述]
- [偏好 2]: [描述]
```

#### lessons.md 格式

```markdown
# [Agent 名称] — 经验教训

> 最后更新: YYYY-MM-DD

## 教训列表

### L-[编号]: [标题]
- **日期**: YYYY-MM-DD
- **项目**: [项目名称]
- **场景**: [什么情况下发生的]
- **教训**: [学到了什么]
- **行动**: [以后怎么做]
- **严重性**: 高/中/低
- **出现次数**: N（同类问题累计出现次数）
- **规则升级**: ⬜ 未升级 / ✅ 已升级为规则 → [规则文件路径]

---
```

### 2.3 教训→规则升级机制（Harness Loop）

> 来源：[Fowler - Harness Engineering](https://martinfowler.com/articles/harness-engineering.html) — 人类不是直接改代码，而是改 Harness 本身。问题反复出现时，改进前馈/反馈控制。

**核心思想**：教训不应停留在文字——反复出现的教训必须变成可执行的规则。

**升级触发条件**（满足任一即触发）：
1. 同类教训**出现次数 ≥ 2**
2. 教训严重性为**高**且行动明确

**升级流程**：

```
lessons.md 中的教训
       │
       ▼
┌──────────────────────┐
│ 触发检查：            │
│ - 出现次数 ≥ 2？      │
│ - 严重性=高 且行动明确？│
└──────────┬───────────┘
           │
     触发 ──┤── 未触发 → 继续积累
           │
           ▼
┌──────────────────────┐
│ 锋/矩审查教训         │
│ 决定升级为什么：       │
│ - Skill 约束条件      │
│ - .mdc 规则文件       │
│ - SOUL.md 新增条目    │
└──────────┬───────────┘
           │
           ▼
    写入对应 Agent 的 rules/ 或 SKILLS.md
    更新 lessons.md 中的「规则升级」字段为 ✅
```

**升级产出类型**：

| 教训类型 | 升级为 | 写入位置 | 审批人 |
|---------|--------|---------|--------|
| 编码规范类 | `.mdc` 规则文件 | `[Agent]/rules/` | 矩 |
| 流程类 | PROTOCOL.md 补充条款 | `PROTOCOL.md` | 锋 |
| 审查遗漏类 | Skill 约束条件 | `[Agent]/SKILLS.md` | 矩 |
| 战略方向类 | SOUL.md 认知模式 | `[Agent]/SOUL.md` | 锋 |

**升级责任人**：
- 技术类教训 → **矩**负责升级
- 战略/流程类教训 → **锋**负责升级
- 任何 Agent 发现教训重复出现时，应主动建议升级

#### knowledge.md 格式

```markdown
# [Agent 名称] — 领域知识

> 最后更新: YYYY-MM-DD

## 知识条目

### K-[编号]: [标题]
- **日期**: YYYY-MM-DD
- **来源**: [从哪个项目/讨论中获得]
- **内容**: [知识描述]
- **适用场景**: [什么时候有用]

---
```

### 2.2 中期记忆（项目级）

存储在项目根目录的 `memory/` 下：

```
project-root/memory/
├── context.md          # 项目上下文（背景、目标、约束）
├── decisions.md        # 决策记录汇总
├── changelog.md        # 需求/方向变更记录
└── retrospective.md    # 项目复盘（项目结束时生成）
```

#### context.md 格式

```markdown
# [项目名称] — 项目上下文

> 最后更新: YYYY-MM-DD

## 项目背景

[项目的来龙去脉，为什么要做这个项目]

## 核心目标

1. [目标 1]
2. [目标 2]

## 关键约束

- 时间: [时间约束]
- 资源: [资源约束]
- 技术: [技术约束]

## 目标用户

[具体的用户画像]

## 核心假设

1. [假设 1] — 验证状态: ✅/❌/⬜
2. [假设 2] — 验证状态: ✅/❌/⬜

## 技术栈

- 前端: [技术选择]
- 后端: [技术选择]
- 数据库: [技术选择]
- 部署: [部署方式]
```

#### decisions.md 格式

```markdown
# [项目名称] — 决策记录

> 最后更新: YYYY-MM-DD

## 决策列表

[从 STATUS.md 的决策日志中汇总，按时间倒序]

### DR-[编号]: [标题]
- **日期**: YYYY-MM-DD
- **决策者**: [Agent]
- **结论**: [选择了什么]
- **理由**: [为什么]
- **影响**: [影响了什么]
```

## 三、读写规范

### 3.1 写入规则

#### 何时写入长期记忆

| 触发条件 | 写入什么 | 写入到哪里 |
|---------|---------|-----------|
| 用户明确表达偏好 | 偏好记录 | `[Agent]/memory/preferences.md` |
| 项目复盘发现教训 | 教训记录 | `[Agent]/memory/lessons.md` |
| 发现可复用的知识 | 知识条目 | `[Agent]/memory/knowledge.md` |
| 用户说"记住这个" | 对应类型 | 对应文件 |

#### 何时写入中期记忆

| 触发条件 | 写入什么 | 写入到哪里 |
|---------|---------|-----------|
| 项目启动 | 项目上下文 | `memory/context.md` |
| 做出重要决策 | 决策记录 | `memory/decisions.md` |
| 需求/方向变更 | 变更记录 | `memory/changelog.md` |
| 项目结束 | 复盘报告 | `memory/retrospective.md` |

#### 写入格式要求

1. **追加写入**：新记录追加到文件顶部（最新在前）
2. **更新时间戳**：每次写入更新文件头部的"最后更新"时间
3. **不删除旧记录**：旧记录保留，通过归档管理
4. **简洁具体**：每条记忆用 1-3 句话说清楚，不写废话

### 3.2 读取规则

#### 何时读取记忆

| 场景 | 读取什么 |
|------|---------|
| Agent 被激活时 | 读取自己的 `memory/` 下所有文件 |
| 进入新项目阶段时 | 读取项目 `memory/context.md` + `decisions.md` |
| 做决策前 | 读取 `lessons.md` 检查是否有相关教训 |
| 与用户交互时 | 读取 `preferences.md` 适配沟通风格 |

#### 读取优先级

```
项目中期记忆（最相关）> Agent 长期记忆（经验）> 其他 Agent 的记忆（参考）
```

### 3.3 归档规则

| 条件 | 操作 |
|------|------|
| 项目完成 | 中期记忆归档到 `memory/archive/[项目名]/` |
| 教训超过 50 条 | 归纳合并，保留 Top 20 最有价值的 |
| 知识过时 | 标记为 `[DEPRECATED]`，不删除 |
| 偏好变更 | 更新旧偏好，保留变更历史 |

## 四、Memory 沉淀流程

### 4.1 会话结束沉淀（推荐）

每次有意义的会话结束时，Agent 应主动检查是否有值得沉淀的信息：

```
会话结束
   │
   ▼
┌──────────────────────┐
│ 自检：本次会话有没有   │
│ 1. 新的用户偏好？      │
│ 2. 值得记录的教训？    │
│ 3. 可复用的知识？      │
│ 4. 重要的决策？        │
└──────────┬───────────┘
           │
     有 ───┤──── 无 → 结束
           │
           ▼
┌──────────────────────┐
│ 写入对应的 memory 文件 │
│ 更新时间戳             │
└──────────────────────┘
```

### 4.2 阶段结束沉淀（强制）

每个项目阶段结束时，主导 Agent **必须**更新项目中期记忆：

| 阶段结束 | 主导 Agent | 必须更新 |
|---------|-----------|---------|
| 构思完成 | 锋 | `context.md`（项目背景和目标） |
| 需求定义完成 | 枢 | `context.md`（用户画像、核心假设）+ `decisions.md` |
| 设计完成 | 绘+矩 | `context.md`（技术栈）+ `decisions.md` |
| 测试完成 | 鉴 | `decisions.md`（质量决策） |
| 发布完成 | 枢 | `changelog.md`（发布记录） |
| 项目结束 | 锋 | `retrospective.md`（复盘报告） |

### 4.3 复盘沉淀（项目结束）

项目结束时，锋主导生成复盘报告，所有参与 Agent 贡献各自领域的教训：

```markdown
# [项目名称] — 复盘报告

## 项目概要
- 目标: [是否达成]
- 时间: [计划 vs 实际]
- 质量: [Bug 数、安全问题数]

## 各角色教训

### 锋（战略）
- [教训]

### 枢（产品）
- [教训]

### 矩（架构）
- [教训]

### 绘（设计）
- [教训]

### 鉴（测试）
- [教训]

### 盾（安全）
- [教训]

## 可复用的模式
1. [模式描述] → 沉淀到 [Agent]/memory/knowledge.md

## 要避免的错误
1. [错误描述] → 沉淀到 [Agent]/memory/lessons.md
```

## 五、质量指标记忆口径

Agent 的质量指标必须能从产物或复盘中追溯，不只写目标。

### 5.1 砺·合伙人 Office Hours 指标

| 指标 | 写入位置 | 字段 |
|------|----------|------|
| 追问深度 | Office Hours 设计文档 / 交接块 | `follow_up_count` |
| 伪需求拦截率 | `合伙人_砺/memory/lessons.md` 或项目复盘 | `blocked_before_dev: true/false` |
| 替代方案采纳率 | 设计文档 + 后续 PRD 对比 | `recommended_path_adopted: true/false` |
| 二评采纳率 | 设计文档 `Second Opinion` | `accepted_items / total_items` |
| Spec 一次通过率 | `Spec Review Result` | `review_rounds` |
| 否决准确率 | 项目复盘 | `veto_outcome` |

### 5.2 写入规则

- 指标不要在每次普通对话里强行写 memory；只在阶段结束、复盘、或 Deep 模式产物完成后写。
- 指标记录要附带产物路径，避免无法追溯。
- 如果指标暴露流程问题，应升级到 `rules/`、`SKILLS.md` 或 `PROTOCOL.md`。

## 六、初始化指南

### 5.1 为现有 Agent 初始化 Memory

每个 Agent 的 `memory/` 目录下创建三个空模板文件：

```bash
# 为每个 Agent 创建 memory 文件
for agent in CEO_锋 PM_枢 QA_鉴 安全_盾 财务_算 顾问_隐 合伙人_砺 架构_矩 内容_辞 设计_绘; do
  touch "$agent/memory/preferences.md"
  touch "$agent/memory/lessons.md"
  touch "$agent/memory/knowledge.md"
done
```

### 5.2 为新项目初始化 Memory

```bash
mkdir -p memory
touch memory/context.md
touch memory/decisions.md
touch memory/changelog.md
```
