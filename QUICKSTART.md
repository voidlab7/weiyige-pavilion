## 一、维弈阁是什么？

维弈阁是一个**13 角色 AI Agent 团队**。你是唯一决策者，Agent 是你的专业团队。

| 快捷指令 | Agent | 一句话职责 |
|---------|-------|-----------|
| `@锋` / `@ceo` | CEO_锋 | 方向对不对？做不做？ |
| `@砺` / `@devil` | 合伙人_砺 | 质疑挑战、Office Hours、值不值得做 |
| `@隐` / `@advisor` | 顾问_隐 | 多角度深度分析、独立二评 |
| `@枢` / `@pm` | PM_枢 | 需求拆解、写 PRD、排期 |
| `@辞` / `@content` | 内容_辞 | 文案创作、去 AI 味 |
| `@寻` / `@scout` | 探索_寻 | 热点监控、趋势研判、竞品调研 |
| `@矩` / `@arch` | 架构_矩 | 技术方案、工程审查 |
| `@绘` / `@design` | 设计_绘 | UI/UX、视觉草图、设计审查 |
| `@铸` / `@dev` / `@forge` | 开发_铸 | 代码实现、Bug 修复、左移检查 |
| `@鉴` / `@qa` | QA_鉴 | 找 Bug、质量测试、回归验证 |
| `@盾` / `@sec` | 安全_盾 | 安全审计、威胁建模 |
| `@算` / `@cfo` | 财务_算 | 成本分析、预算管理、ROI |
| `@启` / `@steward` / `@auto` / `@team` | 启·执事 | 自动编排全链路，不干活只调度 |

不知道该找谁？直接描述需求，路由器会自动分发到合适角色；需要多角色串联时用 `@启` / `@team`。

## 二、3 分钟上手

### Step 1：选择三档模式

| 模式 | 用法 | 适用场景 | 耗时 |
|------|------|---------|------|
| **Quick** | `@quick @矩 看看这个方案` | 小问题、快速确认 | < 5 分钟 |
| **Standard** | `@枢 帮我写 PRD` | 日常任务（默认） | 5-30 分钟 |
| **Deep** | `@deep @砺 我有个新产品想法` | 重大决策、大版本 | 30 分钟+ |

### Step 2：选择运行方式

| 方式 | 典型口令 | 是否启动 task | 适用场景 |
|------|----------|--------------|---------|
| **新会话直连** | `@寻-live` / `直接扮演寻，不要 task` | 否 | 单角色长任务，主会话不想阻塞 |
| **当前会话 inline** | `@寻-inline` / `直接以寻身份回答` | 否 | 快速角色化回答 |
| **启调度** | `@启` / `@team` / `自动跑完` | 由启决定 | 多角色串联、完整流程、全链路 |

规则：

- 用户明确说“不要 task / 不要 subagent / 直接扮演某角色”时，按直连或 inline，不进入 team 待命。
- 用户说“自动跑完 / 全链路 / 走完流程 / @team / @启”时，交给 `启` 编排。
- 小任务不要强行 team；多角色任务不要靠手工串上下文。

### Step 3：试一下

**场景 A：你有个新想法**

```text
@砺 我有个想法：做一个 AI 驱动的宠物健康管理 App。帮我评估一下。
```

→ 砺先做 Office Hours：上下文收集 → Startup / Builder 分流 → 前提挑战 → 替代路径。

**场景 B：你要写需求**

```text
@枢 帮我写一个用户登录功能的 PRD
```

→ 枢把模糊需求消歧义，产出可执行 PRD。

**场景 C：你要写代码**

```text
@铸 根据这个 PRD 实现登录功能，并跑左移检查
```

→ 铸负责实现、最小改动、Lint / 类型 / 快速测试。

**场景 D：你要全链路自动跑**

```text
@team auto 从想法到测试，帮我走完一个新功能流程
```

→ 启创建团队并调度：砺 → 锋 → 枢 → 矩/绘 → 铸 → 鉴 → 盾。

### Step 4：看交接块

每个 Agent 完成实质性工作后，会输出交接块，供下游消费：

```markdown
---
## 📤 交接块（Handoff）

- **来源**: 枢·PM
- **阶段**: 需求定义
- **产出类型**: PRD
- **产物文件**: `docs/prd/example_PRD_v1.md`
- **状态**: 通过 / 有条件通过 / 未通过 / 需要信息
- **关键决策**:
  1. MVP 只做登录，不做注册扩展
- **开放问题**:
  1. 是否需要第三方登录？
- **下游建议**: 建议交 @矩 做架构评审
- **阻塞项**: 无
---
```

## 三、常见使用场景

### 场景 0：新会话直连角色（不阻塞主会话）

```text
@寻-live
不要启动 task，不要创建 subagent，不要进入 team 待命，不要等待 Leader。
请直接以「寻·探索」身份工作。

任务：调研小红书上 AI 图片高清/超分/老照片修复相关需求。
```

### 场景 1：从零启动一个新项目（手动串联）

```text
第 1 步：@砺 我有个想法：[描述想法]。帮我质疑一下。
第 2 步：@锋 砺觉得可以，帮我做战略审查。
第 3 步：@枢 方向确定了，帮我写 PRD。
第 4 步：@矩 PRD 写好了，帮我做架构评审。
第 5 步：@绘 帮我做 UI/UX 方案或设计审查。
第 6 步：@铸 按 PRD 和技术方案实现。
第 7 步：@鉴 代码写完了，帮我测试。
第 8 步：@盾 做安全审计。
第 9 步：发布。
```

### 场景 2：从零启动一个新项目（自动串联）

```text
@team auto 我有个产品想法：[描述想法]。请从评估、需求、设计、开发、测试走完整流程。
```

### 场景 3：写一篇公众号文章

```text
@辞 帮我写一篇关于 [主题] 的公众号文章，目标读者是 [人群]。
```

### 场景 4：审查现有代码

```text
@矩 帮我审查 src/ 目录下的架构设计。
```

### 场景 5：修 Bug

```text
@铸 修复这个报错：[错误信息]。改完后跑左移检查。
```

### 场景 6：需要独立二评

```text
@隐 对这份设计文档做独立二评：找盲点、反例、替代解释和长期后果。
```

## 五、最佳实践

### 推荐做法

1. 小任务用 Quick，不要每次都走全流程。
2. 新想法先给砺，不要跳过质疑环节。
3. 方向拍板找锋，深度二评找隐，自动串联找启。
4. 写代码找铸；矩负责方案和审查，不负责亲自实现。
5. 跟着交接块走，避免下游猜上下文。
6. Deep 模式的文档要落盘，并接受 Spec Review Loop。

### 避免的做法

1. 不要把“我有个想法”直接丢给 PM 写 PRD，先让砺验证需求真实性。
2. 不要忽略 `阻塞项` 和 `开放问题`。
3. 不要手动编辑 memory 文件，除非你正在维护记忆系统本身。
4. 不要在 team 不可用时假装已经多 Agent 协作；应明确报告阻塞。

## 六、FAQ

**Q: 我只有一个小问题，需要走完整流程吗？**  
A: 不需要。用 `@quick @[Agent]`。

**Q: Agent 之间会自动协作吗？**  
A: 普通模式下 Agent 会在交接块建议下一步；你触发下一个角色。`@启` / `@team` 模式下由启自动编排。

**Q: 什么时候用 `@启`？**  
A: 当你想“自动跑完 / 全链路 / 多角色串联 / 从头到尾走流程”时。

**Q: 什么时候用 `@铸`？**  
A: 当任务进入代码实现、Bug 修复、左移检查时。架构方案由矩，具体实现由铸。

**Q: Agent 的记忆会一直保留吗？**  
A: 会。记忆分短期、中期、长期三层，详见 `MEMORY.md`。
