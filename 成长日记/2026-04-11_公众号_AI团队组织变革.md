# 我给 AI 团队做了一次"组织变革"——一个人 + 10 个 AI Agent 的协作实验

> 10 个 AI 员工，我却成了最累的那个人。

---

## 先说结论

我搭了一个叫"维弈阁"的多角色 AI Agent 团队。

10 个 Agent，各司其职：CEO 把方向、PM 拆需求、架构师画蓝图、设计师审 UI、QA 找 Bug……

听起来挺美的。

直到今天，我用第一性原理 + 芒格的多元思维模型，对这个系统做了一次体检。

**结论：病得不轻。**

---

## 问题出在哪

10 个 Agent，本质上是 10 个孤岛。

它们之间没有共享上下文的协议——CEO 批了方案，PM 不知道；架构师说"不行"，PM 说"必须做"，没人知道听谁的。

更要命的是：**我自己成了"人肉路由器"**。

每次切换 Agent，都要手动复制粘贴上下文，手动告诉下一个 Agent "上一个说了什么"。

这不是 10x 效率。这是 0.1x 效率。

---

## 第一性原理：协作的本质是什么

我问了自己一个问题：

> **协作的本质是什么？**

想了一会儿，答案就两个词：**信息的可靠传递** + **决策的明确归属**。

一个真实的团队之所以能协作，不是因为每个人都很厉害，而是因为：
1. A 说的话，B 能准确收到
2. 谁负责什么、谁拍板什么，清清楚楚
3. 审查不是走过场——你提了意见，对方必须回应，改完了还要再看

我的 AI 团队，这三条一条都没做到。

---

## 芒格的五个思维模型，帮我找到了所有问题

### 🔍 逆向思考："如果这个项目失败了，最可能的原因是什么？"

我列了一下：

1. **用户认知负荷太高** — 10 个 Agent，不知道什么时候该用谁
2. **Agent 间信息损耗** — 每次切换都在丢上下文
3. **过度设计** — 150 条认知规则，没有人能记住
4. **从来没真正跑过** — 所有 Agent 的记忆目录都是空的

最后一条让我愣了一下。

所有记忆目录，全是空的。说明这套系统从来没在真实项目里跑过。

### 🎯 能力圈模型："待在你懂的领域里"

我翻了一下 CEO 的配置文件。

里面写满了设计原则——"层级即服务"、"减法默认"……

这些明明是设计师的活。

PM 的配置里包含技术可行性评估，这是架构师的领域。

**每个 Agent 都在越界，没人守住自己的能力圈。**

### 💰 复利思维："什么东西会越用越值钱？"

答案是**记忆**。

如果 Agent 能记住上次的决策、踩过的坑、用户的偏好，那每次使用都在积累价值。

但现实是——所有记忆目录都是空的。每次对话都是失忆状态。

### ⚠️ 激励机制："给我看激励机制，我就能预测结果"

"魔鬼代言人"有一票否决权，CEO 有最终决策权。

但这只是文字约定，没有流程保障。

就像公司章程写得再好，没有董事会流程，也是一纸空文。

### 🗑️ 沉没成本："是否真的需要 10 个 Agent？"

说实话，有些角色高度重叠。战略顾问和魔鬼代言人都在"质疑决策"；财务分析师在个人项目里价值存疑。

但今天我决定先不动刀——**先把协作跑通，再考虑精简。**

---

## 一天的手术：11 项改造

想清楚问题后，我花了一整天，对系统做了一次改造。

### 🔴 修复致命缺陷

**1. 建立协作协议层**

定义了 Agent 之间的"语言"：
- **交接块**：每个 Agent 输出完必须附带标准化的交接信息（来源、状态、关键决策、下游建议）
- **RACI 矩阵**：谁负责、谁审批、谁咨询、谁知会，一张表说清楚
- **冲突解决链**：用户 > CEO > 魔鬼代言人否决权 > 各领域负责人
- **项目状态文件**：所有 Agent 共享一个 STATUS.md，知道"我们在哪"

**2. 实现记忆系统**

设计了三层记忆架构：
- **短期记忆**：当前对话（自动）
- **中期记忆**：项目级（每个阶段结束时沉淀）
- **长期记忆**：团队级（跨项目的教训和偏好）

每个 Agent 有三个记忆文件：偏好、教训、知识。项目有四个记忆文件：上下文、决策、变更、复盘。

这是复利的起点。

**3. 设计反馈闭环**

把所有单向的"审查→报告→结束"改成了闭环：

```
提交 → 审查 → 结果判定
                 │
    ✅ 通过 ──→ 下一阶段
    ⚠️ 有条件 → 改完确认即可
    ❌ 未通过 → 改完重审（最多 3 轮）
    ❓ 缺信息 → 补充后继续
```

关键设计：**迭代次数上限**。CEO 审查最多 3 轮，超了就升级到用户决策。防止 AI 陷入无限循环。

### 🟡 重要改进

**4. 智能路由器**

用户不再需要记住 10 个 Agent 的名字。路由器根据意图自动分发：

- 说"这个方向对吗" → 自动找 CEO
- 说"写个 PRD" → 自动找 PM
- 说"有什么风险" → 自动找魔鬼代言人

还支持快捷指令：`@锋` `@pm` `@arch`，想精确调用也行。

**5. 清理职责越界**

把 CEO 配置里的设计原则还给了设计师，把 PM 配置里的技术评估还给了架构师。

每个人守好自己的能力圈。

### 🟢 体验打磨

**6-11. 六项细节**

- 每个 Agent 增加 **Quick/Standard/Deep 三档模式**（小事不走全流程）
- 修复财务分析师的**硬编码定价**（绝对价格改相对定价）
- 创建**快速入门教程**（5 分钟上手）
- 实现所有**标记为"待装载"的技能**（消除画饼）
- 为所有配置文件增加**版本管理**（可追溯变更）
- 更新项目看板状态

---

## 今天最大的收获

不是这 11 项改造本身。

是一个认知：

> **工具的价值不在于它有多强大，而在于工具之间能否协作。**

10 个顶级专家，如果不能有效沟通，不如 3 个能紧密协作的普通人。

这个道理在人类团队里是常识。但在 AI Agent 领域，大家还在疯狂堆角色、堆能力、堆认知模式。

**没有人在意"协议层"。**

就像 TCP/IP 协议之于互联网——没有人觉得协议很酷，但没有协议，所有的节点都是孤岛。

## 下一步

芒格说：**先在小范围内证明模式有效，再扩展。**

所以下一步不是继续加功能，而是：

> 选一个真实的小项目，只用 3 个核心 Agent（CEO + PM + 架构师），从想法到交付跑通一个完整闭环。

跑通了，记忆目录里就有了第一批真实数据。
跑不通，就知道哪里还需要改。

理论再完美，不如一次实战。

---

## 如果你也在玩多 Agent 协作

我的建议是：

1. **先别急着加角色**，3 个够用就别搞 10 个
2. **先定协议再定能力**，Agent 之间怎么传信息，比 Agent 本身多厉害更重要
3. **给 Agent 记忆**，没有记忆的 AI 只是一次性工具
4. **设计反馈闭环**，单向输出的审查是自欺欺人
5. **逆向思考**，先想"怎么会失败"，再想"怎么能成功"

复杂性是最大的敌人。

芒格说得对——**避免愚蠢，比追求聪明更重要。**
