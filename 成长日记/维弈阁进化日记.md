# 维弈阁进化日记

> 一个人 + AI = 一个团队。这是维弈阁从种子到大树的完整成长记录。

---

## Day 0 — 2026-04-11（上午）· 种子：gstack 启发，10 角色诞生

### 背景

三个星期前，Y Combinator CEO Garry Tan 开源了 gstack——一套工程哲学，把 Claude Code 变成虚拟工程团队。我花了三周，把这套东西从 Claude Code 移植到 CodeBuddy：17 个 Git 提交，+32K 代码净增，29 个规则文件，21+ 知识文档。

用了一段时间，感觉挺好——`ship` 一键发布效率拉满，`careful + guard` 防凌晨手误。

但问题慢慢浮出来：**14 个角色，你经常忘记它们的存在。** 得记住哪个规则叫什么名字，手动 @某个规则才会生效。忘了 @guard 就没有安全检查，忘了 @plan-eng 就跳过工程审查。而且角色之间没有协作——CEO 审完计划，PM 不知道；架构师说"不行"，PM 说"必须做"，没人知道听谁的。

**我自己成了"人肉路由器"。**

### 决定重构

散落的规则文件不是团队。gstack 教会了我工程哲学——先搜索再建造、反谄媚、一键发布。这些东西我全部保留，但协作协议、记忆系统、路由机制，得自己造。

于是，**维弈阁**诞生了。

名字是家人起的。"维"是维度的维，多角色就是多维度；"弈"是博弈的弈，角色之间有合作也有质疑；"阁"是场所，所有角色共处的空间。

### 第一刀：砍角色

gstack 有 14 个角色，我先问自己：哪些是在"写代码"，哪些是在"做决策"？gstack 最重要的洞察——**代码是生成物，不是岗位**。前端、后端砍掉，架构师接管。法务砍掉，个人项目不需要。14 → 10。

### v1.0 初始版本

10 个角色正式上线：CEO_锋、PM_枢、架构_矩、设计_绘、QA_鉴、安全_盾、财务_算、内容_辞、顾问_隐、合伙人_砺。每个角色有 IDENTITY.md（身份）+ SOUL.md（灵魂）+ memory/（空目录）。配上 CLAUDE.md 路由中枢 + install.sh 一键安装。

### 诚实自评

10 个 Agent 是 10 个孤岛。没有协作协议、memory 全空、没有路由器、没有反馈闭环、没有版本管理。

但种子已经种下了。

## Day 0 — 2026-04-11（下午）· 体检：第一性原理 + 芒格多元思维

### 给自己的系统做体检

v1.0 刚搭完，我没急着用。我用第一性原理 + 芒格的多元思维模型做了一次工程深度评估。

**根本命题**：一个人能否通过 AI Agent 团队获得 10x 生产效率？

拆解前提：
- ✅ AI 能理解角色化的专业指令
- ✅ 专业化分工能提升效率
- ⚠️ Agent 之间能有效协作 — **最薄弱环节**
- ⚠️ 一个人能有效调度 10 个 Agent — **认知负荷问题**

### 五个思维模型，五个病灶

**逆向思考**——"如果失败了，最可能的原因？"
1. 用户认知负荷太高：10 个 Agent，不知道该用谁
2. Agent 间信息损耗：每次切换丢上下文
3. 过度设计：150 条认知规则，没人能记住
4. 从来没跑过：所有记忆目录全是空的

**能力圈模型**——CEO 配置里写满了设计原则，那是设计师的活。PM 配置里有技术可行性评估，那是架构师的领域。每个 Agent 都在越界。

**复利思维**——记忆是唯一会越用越值钱的东西。但 memory 全空，复利起点是零。

**激励机制**——砺有一票否决权，锋有最终决策权，但只是文字约定，没有流程保障。

**沉没成本**——是否真需要 10 个？先不动刀，**先把协作跑通，再精简。**

### 核心洞察

> 协作的本质是：**信息的可靠传递 + 决策的明确归属**。没有协议层，10 个 Agent 就是 10 个孤岛。

> 不是 10 个独立 Agent，而是 **3 个层 + 1 个路由器**。路由器才是核心。

## Day 0 — 2026-04-11（晚上）· 手术：一天完成 11 项改造

想清楚问题后，花了一整天做手术。

### P0 致命缺陷修复

**1. 协作协议层（PROTOCOL.md）**
- 交接块：每个 Agent 输出完必须附带标准化的交接信息
- RACI 矩阵：谁负责、谁审批、谁咨询、谁知会
- 冲突解决链：用户 > CEO > 魔鬼代言人否决权 > 各领域负责人
- 项目状态文件：STATUS.md，让所有人知道"我们在哪"

**2. 记忆系统（MEMORY.md）**
- 短期：当前对话（自动）
- 中期：项目级（每阶段结束沉淀）
- 长期：团队级（跨项目教训和偏好）

每个 Agent 三个记忆文件：偏好、教训、知识。这是复利的起点。

**3. 反馈闭环**
- 所有单向的"审查→报告→结束"改成闭环
- 不通过就改，改完再审，最多 3 轮
- 超了升级到用户决策，防止 AI 无限循环

### P1 重要改进

**4. 智能路由器（ROUTER.md）**
- 用户不用记 10 个名字
- "这个方向靠谱吗" → 自动找锋
- "帮我拆需求" → 自动找枢
- 支持快捷指令：`@锋` `@pm` `@arch`
- 路由表内联在配置文件里，零冷启动

**5. 职责边界清理**
- CEO 的设计原则还给设计师
- PM 的技术评估还给架构师
- 每个人守好自己的能力圈

### P2 体验打磨

6. Quick/Standard/Deep 三档模式
7. 修复财务硬编码定价
8. 5 分钟快速入门教程
9. 实现所有"待装载"技能
10. SOUL.md 版本管理
11. README 去 AI 味重写（辞出品）+ 安装脚本迭代

### 这天最大的收获

> **工具的价值不在于它有多强大，而在于工具之间能否协作。** 10 个顶级专家，如果不能有效沟通，不如 3 个能紧密协作的普通人。没有人在意"协议层"，但没有协议，所有节点都是孤岛——就像 TCP/IP 之于互联网。

## Day 1 — 2026-04-12 · 开源：多 Agent 模式 + GPL-3.0

### 变更

- 新增 `agents_for_codebuddy/` 目录，适配 CodeBuddy 多 Agent 模式
- 每个 `.md` 文件通过 YAML frontmatter 定义 Agent 的 name/description/model/tools
- Agent 激活后自动读取完整知识库
- 两种模式（规则模式 / 多 Agent 模式）可以共存
- 正式开源：GPL-3.0 LICENSE + CONTRIBUTING.md

### 思考

> 规则模式的 CLAUDE.md 是"单入口路由"——所有请求走同一个 AI 实例，靠 prompt 切角色。多 Agent 模式是"多实例隔离"——每个 Agent 独立会话，有自己的工具权限和模型配置。前者简单通用，后者更强更灵活。

## Day 2 — 2026-04-13 · 标准化：58 张 Skill 卡片 + 新角色"寻"

### 关键转折

这是维弈阁从"人设集合"到"标准化 Harness"的转折点。

最近 AI Agent 圈子流行一个公式：**Agent = Model + Harness**。Model 是大模型本身，Harness 是模型之外的一切——记忆、协议、路由、反馈、上下文。Can.ac 的实验：同一模型只换了 Harness，编码基准分数从 6.7% → 68.3%。十倍差距，模型一样。

汤道生说："大模型 = 发动机，Harness = 线束。同一模型 + 不同 Harness，Terminal Bench 2.0 从 52.8% → 66.5%。"

### 变更

- 11 个角色共 58 张 Skill 卡片标准化，每张包含：触发条件、输入、输出、工具、约束、示例
- **新角色：寻 · 探索**（第 11 个角色）
  - 8 个 Skills：热点雷达、趋势研判、竞品监控、AI前沿追踪等
- Windows 支持（install.ps1）
- README 大改：增加 Skills 全景图、思想脉络

### 实战验证

做了个对比实验。两个空白 CodeBuddy 工程，同一模型 GLM-5，同一句提示词：

> 做一个本地网页，设计一个宠物的 SBTI，结合 sbti.unun.dev 的产品思路。

唯一区别：一个装了维弈阁，一个没装。

- **装了维弈阁**：CEO 审方向 → PM 拆需求 → 设计出 UI → 架构写代码 → QA 浏览器测。从需求到可运行页面，一次通过。
- **没装维弈阁**：AI 直接写代码，首页出来了，结果页白屏。折腾几轮才勉强跑通，粗糙很多。

不是模型不行，是没有人告诉它先做什么后做什么、做完怎么检查。维弈阁做的就是这件事。

### 踩坑

CodeBuddy 首次打开项目读取 CLAUDE.md，但不代表激活了里面的角色系统。需要再问一句才按流程跑。**配置文件只是说明书，不是自动运行的引擎。你得先点火，发动机才会转。**

## Day 3 — 2026-04-14 · 编排：执事 · 启上线

### 从"人肉路由"到"一句话全链路"

维弈阁终于有了自己的调度器。不再需要手动 @枢 → @矩 → @绘 → @鉴 逐个串联。

### 变更

- **新角色：执事 · 启**（第 12 个角色）
  - 定位：自动编排器（AI Harness），不干活只调度
  - 接收用户顶层任务，通过 CodeBuddy `task` 工具自动 spawn 子 Agent
  - 解析交接块（Handoff Block）驱动下一步，直到全链路完成
  - 预设链路：新功能开发、Bug 修复、设计审查、内容创作、安全审计
- 辞新增热文解构 Skill
- 安装脚本修复 + update 模式上线

### 灵感来源

> 启的设计来自 Sigrid Jin 的 AI Harness 理念——Agent 不需要人肉一个个 @，需要一个调度器串起来。启不写代码、不审方案、不做设计，它只做一件事：确保正确的 Agent 在正确的时机被调用。

## Day 4 — 2026-04-15 · 分水岭：Harness 双层约束

### 思想冲击

读到 Martin Fowler 博客上 Birgitta Böckeler 的论文《Harness Engineering》。核心洞察：**Agent = Model + Harness，Harness 分两类——前馈（Guides，提前告诉 Agent 该怎么做）和反馈（Sensors，事后检查做得对不对）。**

维弈阁之前只有前馈（SOUL.md / PROTOCOL.md），没有反馈（计算型检查）。补上这一层，同一模型的成功率就能从 42% 跳到 78%。

### 一次性落地 8 个 Harness 进化项

| 编号 | 改进 | 解决什么 |
|------|------|---------|
| H-01 | 计算型检查前置 | 矩/鉴先跑 Lint/类型检查/测试，再做 LLM 审查 |
| H-02 | 开发阶段左移检查 | 代码生成后立即跑检查，不等审查门 |
| H-03 | 教训→规则自动升级 | 同类 lesson 出现 2 次，自动转化为 .mdc 规则 |
| H-04 | 启串联 Agent 链 | 无人值守的全链路自动执行 |
| H-05 | 多 Agent 并行审查 | 矩+绘并行，总耗时缩短 40%+ |
| H-06 | 驾驭性评估 | 矩新增 Skill——评估项目的 AI 友好度 |
| H-07 | PRD→可测试规格→自动测试 | 枢的 PRD 增加 Given/When/Then，鉴据此自动生成测试 |
| H-08 | Token 成本看板 | 算新增成本报告 Skill |

### 思考

> 这是质量的分水岭。之前 100% 依赖 LLM 互审——本质上是"让学生互相改卷子"。现在有了计算型检查——相当于加了一台阅卷机，先机器扫一遍，再让老师重点审查机器拿不准的地方。

## Day 5 — 2026-04-16 · 智能调度：启 v3.0

### 启的大脑升级

启从"能调度"进化到"智能调度"。

### 变更

- **启 v3.0 三种编排模式**：
  - auto：全链路自动执行，仅失败或需信息时停下
  - confirm：阶段切换时暂停等用户确认
  - step：每个 Agent 后暂停
  - 平台无关——不依赖 CodeBuddy Team 也能工作
- **Team 支持**：启可以利用 CodeBuddy `task` 工具的 `name`/`team_name` 参数并行 spawn Agent
- **产物验证机制**：阶段门禁从"Agent 说通过就通过"升级为"见文件才过门禁"
- **全面审查**：对编排体系做了一次 8.5/10 的审查，发现 3 个断链风险并记录修复方案

### 编排体系审查发现

- 🔴 寻·探索在 CodeBuddy agents 中缺失（影子角色）
- 🔴 开发阶段是黑盒——设计到测试之间缺少代码实现环节
- 🔴 team 模式下启 spawn 自己存在循环依赖风险

### 思考

> 调度器的价值不在于它自己能干什么，而在于它能让其他人更高效地干活。三种模式让用户在"完全信任"和"逐步确认"之间自由选择。

## Day 5 — 2026-04-16（晚间）· 治理：开源工程化 + 提示词瘦身

### 三件事

**1. 清理 GSLB 来源信息**

维弈阁从 GSLB CodeBuddy 工作流中汲取了灵感（artifact-review、knowledge-distillation、分层规则设计等），但开源工程里不应该标注"来源于 XX"。12 个文件中 23 处 GSLB 引用全部清理，改为保留方法论描述、删除来源标注。

**2. agents_for_codebuddy/ 不入仓库**

这个目录是安装产物，不是源码。`install.sh` 支持远程下载，用户 clone 后执行安装脚本即可获取。从 git 跟踪中移除 11 个文件，加入 `.gitignore`。

**3. 提示词瘦身——784 行安全压缩**

对所有 md 文件做了冗余分析。原则：**核心方法论/认知模式不动**（Agent 启动时需要立即加载），只删重复内容和空模板。

| TODO.md | -471 行 | 已完成的 TODO/Phase6/审查报告全部归档删除 |
| PM_枢/SOUL.md | -66 行 | PRD 模板改引用 skill，进度报告精简 |

| 内容_辞/SOUL.md | -19 行 | 去 AI 味速查表改引用已有 skill |

### 关键决策：SOUL.md vs skills/ 的边界

> **SOUL.md 放"大脑"（启动时加载），skills/ 放"工具箱"（按需加载）。**

认知模式、设计原则、审查流程——这些是角色的"大脑"，必须在 Agent 启动时就在 system prompt 里。移到 skills/ 会变成按需加载，AI 可能在没加载的情况下工作，质量不稳定。

模板、速查表、示例——这些是"工具箱"，只在特定场景需要。放在 skills/ 让 AI 按需 read_file 更合理。

- **铸·开发**（第 13 个角色）：代码锻造师，矩画蓝图，铸把蓝图变实物。闭合了"设计→测试"之间的开发断链。

### 新增 Skills

- **artifact-review**：通用审核技能（多维度打分模板）
- **knowledge-distillation**：知识沉淀方法论

## 踩坑总结

1. **先别急着加角色** — 3 个够用就别搞 10 个
2. **先定协议再定能力** — Agent 之间怎么传信息，比 Agent 本身多厉害更重要
3. **代码是生成物不是岗位** — 不需要"前端 Agent"和"后端 Agent"
4. **路由器比角色重要** — 没有路由器，多角色=多混乱
5. **给 Agent 记忆** — 没有记忆的 AI 只是一次性工具
6. **设计反馈闭环** — 单向输出的审查是自欺欺人
7. **计算型检查补反馈** — 全靠 LLM 互审是"学生互相改卷子"
