## GStack 角色的真正本质——从原始 SKILL.md.tmpl 提炼

GStack 的角色**不是"人"，而是"判断力协议"**。每个 SKILL 定义的是：

> **在什么时机，用什么认知模式，按什么流程，产出什么判断。**

核心发现：

### 1. GStack 没有"实现者"角色

所有 28 个 Skill 中，**没有一个**是"写功能代码"的。唯一会改代码的角色（`/qa`, `/design-review`, `/review`, `/investigate`）改的是**修 Bug**和**修设计缺陷**，不是实现功能。

**功能实现是用户+模型直接做的，不需要角色。** 这是 GStack 思想最反直觉的地方。

### 2. 角色之间的信息流是关键

```
/office-hours → 产出设计文档
    ↓
/plan-ceo-review → 读取设计文档 → 产出 CEO 审查报告
    ↓
/plan-design-review → 读取 DESIGN.md → 产出设计维度评分
    ↓
/plan-eng-review → 读取上述所有 → 产出工程审查报告
    ↓
[用户+模型写代码]
    ↓
/review → 读取 diff → Fix-First → 产出审查结果 → gstack-review-log
    ↓
/qa → 打开浏览器 → 测试+修复 → 产出健康分数
    ↓
/ship → 读取 review-log → 发布
    ↓
/canary → 监控生产环境
```

**每个角色的产出 = 下一个角色的输入。** 这不是"人物设定"，是"流水线协议"。

### 3. 六个层次的判断力（从原始文件提炼）

| 判断力层次 | GStack 角色 | 核心输出 | 认知模式数 |
|-----------|-----------|---------|----------|
| **问题定义** | /office-hours | 设计文档 | 6 个逼问 + 反谄媚 |
| **战略方向** | /plan-ceo-review | 审查报告 + CEO plan | 18 种认知模式 |
| **设计意图** | /design-consultation + /plan-design-review + /design-review | DESIGN.md + 0-10 评分 | 12 种认知模式 + 9 条原则 + 10 个 AI 烂俗反模式 |
| **技术约束** | /plan-eng-review + /investigate + /cso | 架构报告 + 调试报告 + 安全报告 | 15 种认知模式 + Iron Law + 15 阶段安全审计 |
| **质量标准** | /review + /qa + /benchmark + /canary | Fix-First 结果 + 健康分数 + 性能基线 | Fix-First 决策引擎 + WTF-likelihood + 7 维度健康评分 |
| **流程编排** | /autoplan + /ship + /land-and-deploy + /document-release + /retro | 全自动审查 + 发布 + 部署 + 文档 + 回顾 | 6 个自动决策原则 |

### 关键升级点（比上一版更深入）

1. **CEO（锋）**：从一行 SOUL → 注入 18 种认知模式 + Startup/Builder 双模式 + 6 个逼问式问题 + 反谄媚规则
2. **设计（绘）**：从"设计师" → 三位一体（设计系统构建 + 计划评审 + 实现审计）+ 三层竞品综合法 + SAFE/RISK + AI 烂俗 10 反模式
3. **架构（矩）**：从"工程审查" → 四合一（架构审查 + Fix-First 代码审查 + Iron Law 调试 + 性能基线）
4. **安全（盾）**：从"安全" → 15 阶段审计流水线 + AI 安全专项 + 22 条硬排除
5. **QA（鉴）**：加入 WTF-likelihood 自控 + 部署后金丝雀监控
6. **执事（墨）**：加入 6 自动决策原则 + CEO→Design→Eng 全串联

**从 14 角色精简到 11 角色，每个角色从"一行 SOUL"升级到"GStack 级方法论深度"。**
