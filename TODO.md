## 🔴 P0 — 致命缺陷（不解决则系统无法真正运转）

### TODO-001: 设计并实现协作协议层（PROTOCOL.md）

- **What**: 定义 Agent 之间的输入/输出格式、触发条件、信息传递规范和冲突解决规则
- **Why**: 当前 10 个 Agent 是 10 个孤岛，没有协作协议，用户被迫充当"人肉路由器"。协作的本质是信息的可靠传递 + 决策的明确归属
- **Pros**: Agent 之间能真正协作；用户认知负荷大幅降低；系统从"工具集"升级为"团队"
- **Cons**: 设计复杂度高；需要所有 Agent 的 SOUL.md 配合修改
- **建议方案**:
  1. 定义统一的 Agent 输出格式（结构化 JSON/YAML 或标准化 Markdown 模板）
  2. 定义 Agent 间的调用协议（谁触发谁、输入什么、输出什么）
  3. 定义冲突解决优先级（锋 > 砺的否决权 > 其他）
  4. 定义项目状态文件（`STATUS.md`），让每个 Agent 知道"我们在哪"
- **验收标准**: 能在不依赖用户手动传递信息的情况下，完成一个从"想法→审查→拆解→实现"的完整流程
- **Depends on**: 无

### TODO-002: 实现 Memory 系统

- **What**: 为每个 Agent 的 `memory/` 目录定义数据结构和读写规范，实现跨会话记忆
- **Why**: 当前所有 `memory/` 目录为空。没有记忆的 Agent 只是一次性指令执行器，无法积累项目上下文，无法从经验中学习
- **Pros**: Agent 能"成长"；项目上下文不丢失；复利效应（芒格复利思维）
- **Cons**: 需要定义 memory 的生命周期管理（何时写入、何时清理、何时归档）
- **建议方案**:
  1. 定义 memory 数据结构（项目记忆、决策记忆、偏好记忆）
  2. 定义读写规范（每次会话结束自动沉淀关键决策）
  3. 定义 memory 的分层：短期（当前任务）→ 中期（当前项目）→ 长期（团队知识库）
- **验收标准**: Agent 在第二次会话中能引用第一次会话的决策和上下文
- **Depends on**: 无

### TODO-003: 设计反馈闭环机制

- **What**: 为审查流程增加"修改→再审"的迭代循环，而非单向输出
- **Why**: 当前所有 Agent 都是单向输出（审查→报告→结束）。没有反馈的系统无法自我纠正（控制论基本原理）
- **Pros**: 审查结果能真正驱动改进；质量持续提升
- **Cons**: 流程变长；需要定义"何时停止迭代"的退出条件
- **建议方案**:
  1. 每个审查流程增加"修改确认"环节
  2. 定义最大迭代次数（防止无限循环）
  3. 定义"通过"的明确标准
- **验收标准**: CEO_锋审查计划后，修改后的计划能自动重新进入审查流程，直到通过
- **Depends on**: TODO-001

## 🟡 P1 — 重要改进（显著提升可用性）

### TODO-004: 增加路由器/调度器机制

- **What**: 设计一个智能路由层，根据用户意图自动分发到正确的 Agent，而非依赖关键词触发
- **Why**: 10 个 Agent 的触发条件依赖关键词匹配（如"用户说'头脑风暴'"），脆弱且认知负荷高
- **Pros**: 用户只需描述需求，系统自动路由；降低使用门槛
- **Cons**: 路由准确性是新的挑战；增加一层抽象
- **建议方案**:
  1. 设计基于意图的路由规则（而非关键词）
  2. 将 10 个 Agent 归为 3 层：战略层（锋+砺+隐）、执行层（枢+辞）、质量层（矩+绘+鉴+盾）
  3. 路由器先判断层，再判断具体 Agent
- **验收标准**: 用户输入自然语言描述，系统能正确路由到对应 Agent，准确率 > 80%
- **Depends on**: TODO-001

### TODO-005: 精简和合并重叠角色

- **What**: 评估并合并职责高度相似的 Agent，从 10 个精简到 6-7 个
- **Why**: 顾问_隐和合伙人_砺职责高度相似（都是质疑和审视决策）；财务_算在个人项目中价值有限；CEO_锋的 SOUL.md 包含设计原则与设计_绘重叠
- **Pros**: 降低系统复杂度；减少用户选择困难；聚焦核心价值
- **Cons**: 合并可能丢失专业深度；需要重新设计合并后的 SOUL.md
- **建议方案**:
  - 考虑合并：隐+砺 → 战略顾问（兼具芒格智慧和魔鬼代言人）
  - 考虑降级：财务_算 → 从独立 Agent 降为 CEO_锋 的一个 skill
  - 清理 CEO_锋 SOUL.md 中的设计原则，归还给设计_绘
- **验收标准**: 合并后的 Agent 能覆盖原有职责，且用户反馈认知负荷降低
- **Depends on**: 无（但建议在 TODO-001 之后执行）

### TODO-006: 清理 Agent 职责边界越界问题

- **What**: 审查并修正各 Agent 的 SOUL.md，消除职责重叠和越界
- **Why**: CEO_锋包含大量设计原则（"层级即服务"、"减法默认"）应属于设计_绘；PM_枢的"产品三角"包含技术可行性评估应属于架构_矩
- **Pros**: 职责清晰；减少 Agent 间冲突；符合芒格"能力圈"模型
- **Cons**: 需要逐个审查和修改 SOUL.md
- **建议方案**:
  1. 建立职责矩阵（RACI），明确每个 Agent 的 Responsible/Accountable/Consulted/Informed
  2. 逐个清理越界内容
- **验收标准**: 每个 Agent 的 SOUL.md 中不包含其他 Agent 主 owned 的职责内容
- **Depends on**: TODO-005

## 🟢 P2 — 优化项目（提升体验和完整度）

### TODO-007: 为每个 Agent 增加 Quick 模式

- **What**: 每个 Agent 增加 Quick/Standard/Deep 三档模式，小任务不走全流程
- **Why**: 当前每个 Agent 都有复杂的多阶段流程（CEO_锋 10 个审查章节、安全_盾 15 阶段审计），小任务也要走全流程，效率低下
- **Pros**: 灵活适配不同规模的任务；提升日常使用效率
- **Cons**: 需要为每个 Agent 定义三档的边界
- **验收标准**: 用户能通过简单指令选择模式，Quick 模式在 2 分钟内完成
- **Depends on**: 无

### TODO-008: 修复财务_算的硬编码定价问题

- **What**: 将 SOUL.md 中的模型定价表从绝对价格改为相对比较或引用外部数据源
- **Why**: 硬编码的价格（如 claude-sonnet-4-6 "高"）会快速过时，是反模式
- **Pros**: 信息不过时；维护成本低
- **Cons**: 相对比较可能不够精确
- **建议方案**: 改为"高/中/低"相对定价 + 注明"请参考各厂商最新定价页面"
- **验收标准**: SOUL.md 中不包含任何会过时的绝对价格数据
- **Depends on**: 无

## 🧠 指导原则（来自评估分析）

> **第一性原理**：协作的本质是信息的可靠传递 + 决策的明确归属。没有协议层，10 个 Agent 就是 10 个孤岛。

> **芒格·能力圈**：先让 3 个 Agent 跑通一个完整的项目闭环，再考虑扩展到 10 个。

> **芒格·逆向思考**：如果这个项目失败了，最可能的原因是——用户认知负荷过高、Agent 间信息损耗、过度设计。

> **芒格·避免愚蠢**：150 条规则没有人能记住。复杂性是最大的敌人。
